# AI Agent 开发面试八股文

## 一、Agent 基础理论

### Agent 定义与分类 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：Agent 是能够感知环境、自主规划并执行动作以达成目标的智能体
**常见面试问法**：
- "请说说什么是AI Agent？"
- "ReAct和Plan-and-Execute有什么区别？"
- "Reflexion机制是如何提升Agent效果的？"
**回答框架**：
1. Agent是一个能够自主感知、决策、行动的智能体，核心在于"自主性"和"目标驱动"
2. 主流架构分类：ReAct（推理-行动交替）、Plan-and-Execute（先规划后执行）、Reflexion（反思迭代）、Multi-Agent（多智能体协作）
3. ReAct适合简单任务实时决策，Plan-and-Execute适合复杂多步任务，Reflexion通过自我反思提升质量
4. 实际项目中通常混合使用，如先用Plan-and-Execute拆解任务，再用ReAct执行子任务
**延伸考点**：Agent核心组件 → Agent vs 传统软件

### Agent 核心组件 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：Agent由感知、规划、记忆、工具、行动五大组件构成闭环
**常见面试问法**：
- "Agent的核心组件有哪些？"
- "感知-规划-记忆-工具-行动循环是怎么运转的？"
- "记忆系统在Agent中起什么作用？"
**回答框架**：
1. Agent核心组件：感知（Perception）、规划（Planning）、记忆（Memory）、工具（Tools）、行动（Action）
2. 运转流程：感知环境输入→检索记忆→规划决策→选择工具→执行行动→更新记忆→继续感知
3. 感知负责接收用户输入和环境状态，规划负责分解任务和制定策略，记忆负责存储和检索历史信息，工具负责与外部世界交互，行动负责输出最终结果
4. 实际应用中，感知可以是用户消息+系统事件，规划由LLM推理完成，记忆分短期（上下文）和长期（向量库），工具是API/代码/浏览器等
**延伸考点**：Agent记忆系统 → 工具使用与集成

### Agent vs 传统软件 [频率: ⭐⭐⭐⭐]
**核心要点**：Agent是目标驱动的自主决策系统，传统软件是规则驱动的确定性系统
**常见面试问法**：
- "Agent和传统自动化脚本有什么区别？"
- "什么时候该用Agent，什么时候该用传统软件？"
**回答框架**：
1. 传统软件基于确定性规则，输入→处理→输出路径固定；Agent基于目标驱动，自主选择路径达成目标
2. 核心差异：自主性（Agent自主决策vs软件按规则执行）、适应性（Agent适应新场景vs软件需重写逻辑）、不确定性（Agent输出非确定性vs软件输出确定性）
3. Agent适合开放域、非结构化、需要灵活决策的场景；传统软件适合流程固定、要求确定性的场景
4. 实际项目中两者结合：Agent处理需要理解和决策的部分，传统软件处理确定性流程
**延伸考点**：Agent涌现能力 → Agent可靠性

### Agent 涌现能力 [频率: ⭐⭐⭐]
**核心要点**：Agent在复杂任务中可能表现出未被显式编程的能力，同时带来不可预测性
**常见面试问法**：
- "什么是Agent的涌现能力？"
- "如何控制Agent的不可预测行为？"
**回答框架**：
1. 涌现能力指Agent在组合使用工具和推理时，表现出超越单一组件能力的现象
2. 不可预测性来源：LLM生成的随机性、工具组合的复杂性、多步推理的误差累积
3. 控制策略：设置Guardrails约束行为边界、关键节点人工审核、输出校验机制、沙箱隔离执行
4. 安全边界设计原则：最小权限原则、可回滚原则、人类可干预原则
**延伸考点**：Prompt注入防御 → Guardrails

## 二、LLM 基础（Agent 相关）

### 大模型推理机制 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：LLM基于自回归逐token生成，KV Cache是推理加速的关键技术
**常见面试问法**：
- "请说说大模型的推理过程"
- "KV Cache的原理是什么？为什么能加速推理？"
- "Temperature、Top-p、Top-k这些采样策略有什么区别？"
**回答框架**：
1. 大模型推理是自回归过程：每次根据已生成的token预测下一个token的概率分布，采样得到新token
2. KV Cache：缓存已计算过的Key和Value矩阵，避免重复计算，将推理复杂度从O(n²)降至O(n)；Prefill阶段计算所有KV，Decode阶段逐token生成并复用KV Cache
3. 采样策略：Temperature控制分布平滑度（低温趋向贪心，高温趋向随机）、Top-k限制候选token数量、Top-p（nucleus sampling）按累积概率截断候选集
4. Agent场景中通常用较低Temperature（0.0-0.3）保证工具调用的稳定性，创意生成场景可适当提高
**延伸考点**：上下文窗口管理 → 模型微调与部署

### Prompt Engineering [频率: ⭐⭐⭐⭐⭐]
**核心要点**：通过精心设计提示词引导LLM输出，CoT是Agent推理的核心技术
**常见面试问法**：
- "CoT和ToT有什么区别？"
- "Few-shot和Zero-shot各自适用什么场景？"
- "如何设计一个好的系统提示词？"
**回答框架**：
1. Prompt Engineering是通过优化输入提示来提升LLM输出质量的技术
2. 核心方法：Zero-shot（直接提问）、Few-shot（提供示例）、CoT（链式思考，分步推理）、ToT（树式思考，探索多条推理路径）
3. CoT让模型"说出思考过程"，显著提升数学、逻辑等复杂推理任务的准确率；ToT在CoT基础上允许回溯和探索，适合需要搜索最优解的任务
4. 系统提示词设计原则：角色定义清晰、任务描述明确、输出格式规范、约束条件明确、提供示例参考
**延伸考点**：Function Calling → Agent核心组件

### Function Calling [频率: ⭐⭐⭐⭐⭐]
**核心要点**：Function Calling是LLM与外部工具交互的标准协议，通过JSON Schema定义工具接口
**常见面试问法**：
- "Function Calling的原理是什么？"
- "如何设计一个好的工具Schema？"
- "并行工具调用是怎么实现的？"
**回答框架**：
1. Function Calling是LLM根据用户意图和可用工具描述，生成结构化工具调用请求的机制
2. 流程：定义工具的JSON Schema（名称、描述、参数）→LLM判断是否需要调用工具→生成工具调用JSON→执行工具→将结果返回LLM→LLM继续推理或生成最终回答
3. Schema设计要点：函数名和描述要清晰无歧义、参数类型和约束要完整、枚举值要穷举、必填/选填要明确
4. 并行调用：当多个工具之间无依赖关系时，LLM可一次生成多个工具调用，减少交互轮次；需注意上下文长度和结果合并
**延伸考点**：工具设计原则 → MCP协议

### 上下文窗口管理 [频率: ⭐⭐⭐⭐]
**核心要点**：Agent长对话中需有效管理上下文，避免超出窗口限制和信息丢失
**常见面试问法**：
- "Agent对话很长时怎么处理上下文？"
- "滑动窗口和上下文压缩有什么区别？"
**回答框架**：
1. 上下文窗口是LLM单次能处理的最大token数，超出会导致信息截断或报错
2. 管理策略：长文本截断（保留最近N轮）、滑动窗口（固定窗口大小滑动）、上下文压缩（用LLM摘要历史对话）、检索增强（只检索相关历史）
3. 截断简单但丢失早期信息，压缩保留语义但有信息损失，检索增强最精准但增加延迟
4. 实际方案：短期记忆用滑动窗口保留最近对话，长期记忆用向量库存储关键信息，关键指令始终放在系统提示词中
**延伸考点**：Agent记忆系统 → RAG基础流程

### 模型微调与部署 [频率: ⭐⭐⭐]
**核心要点**：SFT/RLHF/DPO是主流微调方法，vLLM/TGI是高效推理部署方案
**常见面试问法**：
- "SFT和RLHF有什么区别？"
- "DPO相比RLHF有什么优势？"
- "vLLM为什么比原生transformers快？"
**回答框架**：
1. SFT（监督微调）用标注数据训练模型模仿目标行为；RLHF（基于人类反馈的强化学习）先训练奖励模型再用PPO优化；DPO（直接偏好优化）跳过奖励模型直接用偏好数据优化
2. SFT适合领域适配，RLHF/DPO适合对齐人类偏好；DPO比RLHF训练更稳定、不需要奖励模型
3. 推理部署：vLLM通过PagedAttention管理KV Cache内存、连续批处理提升吞吐；TGI（Text Generation Inference）支持张量并行和流式生成；ollama适合本地开发测试
4. Agent场景选型：对延迟敏感选vLLM+PagedAttention，多模型协作选TGI，开发调试选ollama
**延伸考点**：大模型推理机制 → Agent成本优化

## 三、Agent 框架与工具链

### LangChain / LangGraph [频率: ⭐⭐⭐⭐⭐]
**核心要点**：LangChain提供Agent开发的基础抽象，LangGraph用状态机实现复杂Agent流程
**常见面试问法**：
- "LangChain的核心模块有哪些？"
- "LangGraph和LangChain是什么关系？"
- "LangGraph的状态机怎么设计？"
**回答框架**：
1. LangChain核心模块：Chain（调用链）、Agent（智能体）、Tool（工具）、Retriever（检索器）、Memory（记忆）、Callback（回调追踪）
2. LangChain适合线性流程，复杂分支和循环难以表达；LangGraph基于图的状态机，支持条件分支、循环、并行，适合复杂Agent流程编排
3. LangGraph核心概念：State（共享状态）、Node（处理节点）、Edge（转移边，含条件边）、Checkpoint（状态快照，支持断点续跑和人工审核）
4. 实际项目：简单RAG用LangChain，复杂Agent用LangGraph；LangGraph支持human-in-the-loop、时间旅行调试、子图嵌套
**延伸考点**：CrewAI/AutoGen → MCP协议

### CrewAI / AutoGen [频率: ⭐⭐⭐⭐]
**核心要点**：CrewAI基于角色分工协作，AutoGen基于对话编排多Agent交互
**常见面试问法**：
- "CrewAI和AutoGen有什么区别？"
- "Multi-Agent框架怎么选？"
**回答框架**：
1. CrewAI：以角色（Role）为核心，定义Agent的角色、目标、 backstory，通过Task分配和Process（Sequential/Hierarchical）编排协作
2. AutoGen：以对话（Conversation）为核心，Agent之间通过消息传递协作，支持自定义对话模式、人类代理参与、代码执行
3. CrewAI适合角色明确、流程固定的协作场景；AutoGen适合需要灵活对话、动态决策的场景
4. 选型考虑：任务复杂度（简单用CrewAI，复杂用AutoGen）、人类参与度（AutoGen更灵活）、调试需求（CrewAI更结构化）
**延伸考点**：Multi-Agent架构 → Agent协作模式

### Dify / Coze / FastGPT [频率: ⭐⭐⭐⭐]
**核心要点**：低代码Agent平台通过可视化工作流降低开发门槛
**常见面试问法**：
- "Dify和Coze有什么区别？"
- "低代码平台能做复杂Agent吗？"
**回答框架**：
1. Dify：开源低代码平台，支持工作流编排、RAG、Agent模式，可私有化部署，适合企业级应用
2. Coze：字节跳动出品，集成丰富插件和知识库，支持多平台发布（飞书/微信/Discord），适合快速上线
3. FastGPT：专注知识库问答，Flow模块支持工作流编排，开源可私有化，适合RAG场景
4. 低代码平台适合快速验证和简单场景，复杂逻辑（动态工具选择、多步推理、精细控制）仍需代码开发
**延伸考点**：LangChain/LangGraph → RAG基础流程

### OpenAI Agents SDK [频率: ⭐⭐⭐⭐]
**核心要点**：OpenAI Agents SDK提供轻量级多Agent编排，核心概念是Swarm和Handoff
**常见面试问法**：
- "OpenAI Agents SDK的核心概念是什么？"
- "Handoff机制是怎么工作的？"
**回答框架**：
1. OpenAI Agents SDK是OpenAI官方的Agent开发框架，核心概念：Agent（定义角色和工具）、Handoff（Agent间转交控制权）、Guardrails（输入输出安全约束）
2. Handoff机制：当当前Agent判断需要其他Agent处理时，通过handoff工具将对话控制权转交给目标Agent，目标Agent继承完整对话上下文
3. Guardrails：输入Guardrail在Agent处理前校验用户输入，输出Guardrail在Agent响应前校验输出内容，支持同步和异步校验
4. 适合轻量级多Agent场景，与OpenAI模型深度集成，但对非OpenAI模型支持有限
**延伸考点**：Multi-Agent架构 → Guardrails

### MCP 协议 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：MCP是Anthropic提出的模型上下文协议，标准化LLM与外部工具/资源的交互
**常见面试问法**：
- "MCP协议是什么？解决了什么问题？"
- "MCP和Function Calling有什么区别？"
- "MCP的架构是怎样的？"
**回答框架**：
1. MCP（Model Context Protocol）是开放协议，标准化LLM应用与外部数据源、工具之间的连接方式，类似"AI领域的USB协议"
2. 核心架构：Host（LLM应用，如Claude Desktop）→ Client（协议客户端，与Server一对一连接）→ Server（提供工具、资源、提示模板的服务端）
3. MCP三大能力：Tools（可调用的函数，类似Function Calling）、Resources（可读取的数据源，如文件/数据库）、Prompts（预定义的提示模板）
4. MCP vs Function Calling：Function Calling是单一模型的工具调用接口，MCP是跨应用的标准协议；MCP解决工具碎片化问题，一次开发到处接入；支持工具发现、能力协商
**延伸考点**：Function Calling → 工具设计原则

## 四、RAG 与知识管理

### RAG 基础流程 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：RAG通过检索外部知识增强LLM生成，核心流程是索引→检索→生成
**常见面试问法**：
- "请说说RAG的完整流程"
- "Query改写有什么作用？"
- "混合检索和纯向量检索有什么区别？"
**回答框架**：
1. RAG（Retrieval-Augmented Generation）将信息检索与大模型生成结合，解决LLM知识过时和幻觉问题
2. 完整流程：文档加载→文本切分→Embedding向量化→存入向量库（索引阶段）；用户Query→Query改写/扩展→向量检索+关键词检索（混合检索）→重排序→Top-K文档注入Prompt→LLM生成（检索生成阶段）
3. Query改写：将用户模糊查询改写为更精确的检索词，包括同义扩展、HyDE（假设性文档嵌入）、多查询分解
4. 混合检索结合语义检索（理解意图）和关键词检索（精确匹配），比单一检索方式召回率更高
**延伸考点**：向量数据库 → RAG优化 → Chunk策略

### 向量数据库 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：向量数据库专门存储和检索高维向量，HNSW是主流近似最近邻索引算法
**常见面试问法**：
- "向量数据库和传统数据库有什么区别？"
- "HNSW和IVF索引有什么区别？"
- "Milvus和Chroma怎么选？"
**回答框架**：
1. 向量数据库存储数据的向量表示，支持基于语义相似度的快速检索，传统数据库基于精确匹配
2. 索引算法：HNSW（分层可导航小世界图）查询速度快、召回率高，但内存占用大；IVF（倒排文件索引）通过聚类缩小搜索范围，内存友好但需要调参（nlist/nprobe）
3. 主流产品：Milvus（分布式、生产级、功能丰富）、Pinecone（全托管、易用）、Chroma（轻量、适合开发）、Weaviate（支持多模态、GraphQL查询）
4. 选型：开发原型用Chroma，生产环境用Milvus/Pinecone，需要多模态用Weaviate
**延伸考点**：Embedding模型 → RAG基础流程

### Embedding 模型 [频率: ⭐⭐⭐⭐]
**核心要点**：Embedding模型将文本转为稠密向量，选型需平衡维度、性能和语言支持
**常见面试问法**：
- "BGE和E5有什么区别？"
- "Embedding维度越高越好吗？"
- "中文场景用什么Embedding模型？"
**回答框架**：
1. Embedding模型将文本映射到高维向量空间，语义相似的文本向量距离更近
2. 主流模型：BGE（BAAI出品，中文效果优秀，bge-large-zh-v1.5常用）、E5（微软出品，多语言，需加前缀指令）、text-embedding-3（OpenAI出品，支持动态维度）、Cohere Embed（多语言，性能强）
3. 维度选择：维度越高信息保留越完整但存储和计算成本越高；OpenAI text-embedding-3支持动态降维，可按需平衡效果和成本
4. 中文场景首选BGE系列，多语言场景选E5或Cohere，追求便捷用OpenAI API
**延伸考点**：向量数据库 → Chunk策略

### Chunk 策略 [频率: ⭐⭐⭐⭐]
**核心要点**：文本切分质量直接影响RAG检索效果，需根据文档类型选择合适策略
**常见面试问法**：
- "Chunk大小怎么确定？"
- "语义切分和固定大小切分有什么区别？"
- "Markdown/代码文档怎么切分？"
**回答框架**：
1. Chunk策略决定文档如何被切分成可检索单元，直接影响检索精度和生成质量
2. 固定大小切分：按字符/token数切分，简单但可能切断语义；语义切分：用Embedding计算相邻文本相似度，在语义断裂处切分；递归切分：按分隔符层级递归切分（段落→句子→词）；文档结构切分：按标题/章节/代码块等结构化标记切分
3. Chunk大小：通常500-1000 token，重叠50-100 token；太小丢失上下文，太大引入噪声
4. Markdown按标题层级切分，代码按函数/类切分，表格尽量保持完整，PDF需先做版面分析再切分
**延伸考点**：RAG基础流程 → RAG优化

### RAG 优化 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：RAG优化贯穿检索前、检索中、检索后全链路，高级方案引入自我评估和纠错
**常见面试问法**：
- "RAG效果不好怎么优化？"
- "HyDE和Query2Doc有什么区别？"
- "Self-RAG和CRAG是什么？"
**回答框架**：
1. 检索前优化：Query改写（纠正拼写、补充上下文）、HyDE（让LLM生成假设性答案，用答案向量检索）、Query2Doc（用LLM扩展查询文档）、自查询（从Query提取元数据过滤条件）
2. 检索中优化：混合检索（向量+关键词）、多路召回、调整Top-K
3. 检索后优化：重排序（Cohere Rerank/BGE-Reranker对检索结果精排）、上下文压缩（去除无关片段）
4. 高级方案：CRAG（纠错RAG，检索结果质量差时用网络搜索补充）、Self-RAG（自反思RAG，LLM自我判断是否需要检索、检索结果是否相关、生成是否可靠）
**延伸考点**：RAG基础流程 → Agent记忆系统

## 五、Agent 记忆系统

### 短期记忆 [频率: ⭐⭐⭐⭐]
**核心要点**：短期记忆是Agent当前对话的上下文，受限于LLM上下文窗口
**常见面试问法**：
- "Agent的短期记忆怎么实现？"
- "对话历史太长怎么办？"
**回答框架**：
1. 短期记忆存储当前会话的对话历史和中间状态，直接放在LLM的上下文窗口中
2. 实现方式：消息列表（system/user/assistant/tool消息序列）、滑动窗口（保留最近N轮）、摘要压缩（对早期对话生成摘要）
3. 挑战：上下文窗口有限、长对话信息丢失、关键信息被淹没
4. 优化：关键信息提取后存入长期记忆、系统提示词中放置不可遗忘的指令、对话摘要保留关键决策和结论
**延伸考点**：长期记忆 → 上下文窗口管理

### 长期记忆 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：长期记忆持久化Agent的知识和经验，支持跨会话信息复用
**常见面试问法**：
- "Agent的长期记忆怎么实现？"
- "知识图谱和向量库在记忆系统中各有什么优势？"
- "用户画像怎么持久化？"
**回答框架**：
1. 长期记忆跨会话持久存储，解决短期记忆窗口有限和会话隔离的问题
2. 实现方式：向量存储（语义检索，适合非结构化知识）、知识图谱（关系推理，适合结构化实体关系）、用户画像（结构化存储用户偏好/历史行为）、数据库（结构化数据持久化）
3. 向量库适合"回忆相似经历"，知识图谱适合"推理实体关系"，用户画像适合"个性化服务"
4. 实际方案：用户偏好存画像，对话摘要存向量库，业务知识存知识图谱，三者协同提供完整记忆
**延伸考点**：短期记忆 → 记忆检索与更新

### 记忆检索与更新 [频率: ⭐⭐⭐⭐]
**核心要点**：记忆检索需平衡相关性和时效性，记忆更新需设计遗忘和压缩机制
**常见面试问法**：
- "怎么从大量记忆中检索最相关的？"
- "记忆需要遗忘机制吗？怎么设计？"
- "记忆压缩怎么做？"
**回答框架**：
1. 记忆检索：基于语义相似度检索（向量检索）、基于时间衰减（近期记忆权重更高）、基于重要性评分（关键决策记忆权重更高）、混合检索策略
2. 遗忘机制：时间衰减（随时间降低记忆权重）、重要性过滤（低重要性记忆定期清理）、容量限制（超出阈值触发压缩或删除）
3. 记忆压缩：对多条相关记忆生成摘要、合并重复信息、提取关键事实
4. 更新策略：增量更新（新记忆直接追加）、合并更新（新旧记忆合并去重）、反思更新（LLM审视记忆并修正矛盾）
**延伸考点**：短期记忆 → 长期记忆

## 六、工具使用与集成

### 工具设计原则 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：好的工具设计让LLM准确理解和使用，单一职责+清晰Schema+健壮错误处理
**常见面试问法**：
- "如何设计一个LLM能准确使用的工具？"
- "工具描述怎么写才能让模型正确调用？"
- "工具的错误处理怎么做？"
**回答框架**：
1. 工具设计三大原则：单一职责（一个工具做一件事）、清晰Schema（参数名和描述无歧义）、健壮错误处理（异常信息可被LLM理解并重试）
2. Schema设计：函数名用动词+名词（如search_product）、描述说明功能+适用场景+限制条件、参数提供枚举值和默认值、必填参数最小化
3. 错误处理：返回结构化错误信息（错误码+错误描述+建议操作）、区分可重试错误和不可重试错误、超时和限流自动重试
4. 反模式：一个工具做太多事（LLM容易混淆参数）、描述模糊（导致误调用）、缺少示例（复杂参数难以理解）
**延伸考点**：Function Calling → API集成

### API 集成 [频率: ⭐⭐⭐⭐]
**核心要点**：Agent调用外部API需处理认证、限流、重试等工程问题
**常见面试问法**：
- "Agent调用API怎么处理认证？"
- "API限流怎么应对？"
- "REST和GraphQL在Agent场景怎么选？"
**回答框架**：
1. 认证鉴权：API Key通过环境变量注入（不进Prompt）、OAuth2.0流程由工具层处理（LLM不接触token）、敏感操作需二次确认
2. 限流应对：指数退避重试、请求队列管理、多账号轮询、降级策略（限流时返回友好提示而非报错）
3. REST vs GraphQL：REST接口固定、Schema简单、LLM更容易正确调用；GraphQL灵活但Schema复杂、LLM容易构造错误查询；Agent场景优先选REST
4. 工程实践：统一封装HTTP客户端、超时设置、熔断机制、请求/响应日志记录
**延伸考点**：工具设计原则 → 代码执行

### 代码执行 [频率: ⭐⭐⭐⭐]
**核心要点**：Agent执行代码必须在沙箱中隔离运行，防止安全风险
**常见面试问法**：
- "代码解释器的安全怎么做？"
- "沙箱执行怎么实现？"
**回答框架**：
1. Agent生成的代码必须在沙箱中执行，防止恶意代码影响宿主系统
2. 沙箱方案：Docker容器（资源隔离+网络隔离+文件系统隔离）、gVisor（用户态内核，更强隔离）、WebAssembly（轻量级沙箱）、Jupyter Kernel（适合数据分析场景）
3. 安全措施：禁止网络访问（或白名单）、限制CPU/内存/执行时间、只读文件系统（指定目录可写）、禁止系统调用
4. 实际方案：Docker+资源限制+网络白名单是最常用组合，代码执行超时自动kill，输出截断防止内存溢出
**延伸考点**：API集成 → Agent安全与评估

### 浏览器自动化 [频率: ⭐⭐⭐]
**核心要点**：浏览器自动化让Agent能操作网页，Computer Use是更通用的GUI交互方式
**常见面试问法**：
- "Playwright和Selenium有什么区别？"
- "Computer Use是什么？"
**回答框架**：
1. 浏览器自动化让Agent能浏览网页、填写表单、点击按钮、提取信息
2. Playwright vs Selenium：Playwright现代API设计、自动等待、多浏览器支持、更适合Agent场景；Selenium生态成熟但API较旧、需要显式等待
3. Computer Use：Anthropic提出的通用GUI交互方式，模型直接"看"屏幕截图并输出鼠标/键盘操作，不依赖DOM解析，可操作任意桌面应用
4. Agent场景：结构化网页用Playwright（精准、高效），非结构化界面用Computer Use（通用但慢且贵）
**延伸考点**：工具设计原则 → API集成

## 七、Multi-Agent 系统

### Multi-Agent 架构 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：Multi-Agent架构决定Agent间的组织关系和通信方式
**常见面试问法**：
- "Multi-Agent有哪些架构模式？"
- "中心化和去中心化架构怎么选？"
- "Agent之间怎么通信？"
**回答框架**：
1. 三种主流架构：中心化（Orchestrator统一调度）、去中心化（Agent平等协作）、层级式（Manager-Worker分层管理）
2. 中心化架构：一个主Agent负责任务分配和结果汇总，其他Agent只执行分配的任务；优点是控制力强、流程清晰，缺点是单点瓶颈
3. 去中心化架构：Agent之间平等通信、自主协作；优点是灵活、无单点故障，缺点是协调困难、可能死锁
4. 层级式架构：Manager拆解任务→Sub-manager进一步分解→Worker执行；适合复杂大型任务，兼顾控制和灵活性
5. 通信协议：消息传递（结构化JSON）、共享状态（黑板模式）、事件驱动（发布订阅）
**延伸考点**：Agent协作模式 → Agent冲突与一致性

### Agent 协作模式 [频率: ⭐⭐⭐⭐]
**核心要点**：不同协作模式适合不同任务类型，分工/讨论/投票/流水线各有适用场景
**常见面试问法**：
- "Multi-Agent有哪些协作模式？"
- "投票模式怎么实现？"
**回答框架**：
1. 分工模式：每个Agent负责不同子任务，并行执行后合并结果；适合可独立拆解的任务（如同时搜索多个数据源）
2. 讨论模式：Agent围绕同一问题展开多轮对话，互相质疑和补充；适合需要多角度思考的任务（如方案评审）
3. 投票模式：多个Agent独立给出答案，通过投票/加权平均/多数决得出最终结果；适合需要降低错误率的任务（如代码审查）
4. 流水线模式：Agent按顺序处理，前一个的输出是后一个的输入；适合有明确处理步骤的任务（如需求分析→设计→编码→测试）
**延伸考点**：Multi-Agent架构 → Agent冲突与一致性

### Agent 冲突与一致性 [频率: ⭐⭐⭐]
**核心要点**：多Agent协作需处理资源竞争、决策冲突和状态一致性问题
**常见面试问法**：
- "多个Agent同时修改同一数据怎么办？"
- "Agent之间意见不一致怎么处理？"
**回答框架**：
1. 冲突类型：资源冲突（多个Agent同时访问同一资源）、决策冲突（Agent给出矛盾建议）、状态冲突（共享状态不一致）
2. 资源冲突：加锁机制（分布式锁）、队列串行化、资源分区（每个Agent操作不同分区）
3. 决策冲突：优先级机制（高优先级Agent决策优先）、投票机制（多数决）、仲裁者（专门Agent做最终裁决）
4. 状态一致性：共享状态存储（所有Agent读写同一数据源）、事件溯源（通过事件日志重建状态）、最终一致性（允许短暂不一致，通过同步机制收敛）
**延伸考点**：Multi-Agent架构 → Agent协作模式

## 八、Agent 安全与评估

### Prompt 注入防御 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：Prompt注入是Agent最常见的安全威胁，需多层防御
**常见面试问法**：
- "什么是Prompt注入？怎么防御？"
- "越狱攻击有哪些类型？"
- "如何防止Agent泄露敏感数据？"
**回答框架**：
1. Prompt注入：攻击者通过用户输入注入恶意指令，覆盖或篡改Agent的原始指令
2. 攻击类型：直接注入（用户输入中嵌入恶意指令）、间接注入（通过外部数据源注入，如网页/文档中隐藏指令）、越狱（绕过安全限制）、数据泄露（诱导Agent输出系统提示词或敏感数据）
3. 防御策略：输入输出分离（用户输入和系统指令用不同标记隔离）、输入校验（检测可疑模式）、权限最小化（工具只授予必要权限）、输出过滤（检测敏感信息泄露）
4. 深度防御：多层防御叠加，没有单一方案能完全防御；关键操作需人工确认；定期红队测试
**延伸考点**：Guardrails → Agent可观测性

### Agent 可观测性 [频率: ⭐⭐⭐⭐]
**核心要点**：Agent行为不可预测，完善的可观测性是调试和优化的基础
**常见面试问法**：
- "Agent的可观测性怎么做？"
- "LangSmith能解决什么问题？"
- "Agent的Trace怎么设计？"
**回答框架**：
1. Agent可观测性三大支柱：Trace（追踪完整调用链）、日志（记录关键事件）、指标（量化性能和成本）
2. Trace设计：记录每次LLM调用的输入输出、工具调用的参数和结果、决策节点的推理过程；形成完整的调用树
3. 工具：LangSmith（LangChain生态，可视化Trace和调试）、Arize Phoenix（开源，支持多框架）、LangFuse（开源，功能全面）
4. 实践：每次用户请求生成唯一trace_id、关键节点打标（如planning/tool_call/error）、记录token消耗和延迟、异常链路自动告警
**延伸考点**：Agent评估 → Agent线上故障排查

### Agent 评估 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：Agent评估需多维度量化，任务完成率是核心指标
**常见面试问法**：
- "Agent怎么评估效果？"
- "工具使用准确率怎么衡量？"
- "如何构建Agent的评估数据集？"
**回答框架**：
1. 评估维度：任务完成率（最终结果是否正确）、工具使用准确率（是否选对工具、参数是否正确）、延迟（端到端响应时间）、成本（token消耗和API调用费用）
2. 评估方法：人工评估（最准确但成本高）、LLM-as-Judge（用强模型评估弱模型输出）、自动化测试（预定义输入输出对）、A/B测试（线上对比不同方案）
3. 评估数据集构建：收集真实用户case、人工标注golden answer、覆盖正常和边界场景、持续迭代更新
4. 实践：建立评估基准线→每次改动跑评估→关注回归→持续优化
**延伸考点**：Agent可观测性 → Agent可靠性

### Guardrails [频率: ⭐⭐⭐⭐]
**核心要点**：Guardrails在Agent输入输出层面设置安全护栏，约束Agent行为边界
**常见面试问法**：
- "Guardrails怎么实现？"
- "输入输出过滤怎么做？"
- "如何约束Agent不执行危险操作？"
**回答框架**：
1. Guardrails是Agent安全的前置防线，在LLM处理前和输出后进行校验
2. 输入Guardrails：检测Prompt注入、过滤敏感信息、验证输入格式、拒绝超出能力范围的请求
3. 输出Guardrails：检测有害内容、过滤敏感数据泄露、验证输出格式、约束行为范围（如禁止执行删除操作）
4. 实现方式：规则引擎（正则/关键词匹配，快速但死板）、LLM校验（用另一个LLM判断安全性，灵活但增加延迟和成本）、分类模型（训练专用安全分类器，平衡速度和准确率）
**延伸考点**：Prompt注入防御 → Agent安全与评估

## 九、高频项目方向

### 智能客服 Agent
**核心要点**：RAG+工具调用+多轮对话的典型落地场景
**常见面试问法**：
- "智能客服Agent怎么设计？"
- "如何保证客服Agent不胡说？"
**回答框架**：
1. 架构：用户输入→意图识别→RAG检索知识库→工具调用（查订单/退款等）→LLM生成回答
2. 关键技术：RAG保证回答基于知识库（减少幻觉）、工具调用实现业务操作、多轮对话管理上下文、Guardrails防止越界回答
3. 难点：知识库更新及时性、多轮对话状态管理、复杂意图理解、情绪识别与安抚
4. 优化：用户反馈闭环优化知识库、常见问题缓存、人工接管机制、对话质量评估

### 编程助手 Agent
**核心要点**：代码生成+调试+项目理解的端到端开发辅助
**常见面试问法**：
- "编程助手Agent怎么实现？"
- "如何让Agent理解整个项目？"
**回答框架**：
1. 架构：用户需求→代码检索（项目上下文）→代码生成/修改→沙箱执行验证→调试修复→输出结果
2. 关键技术：项目代码索引（AST+向量检索）、代码生成（Function Calling调用文件操作工具）、沙箱执行验证、错误反馈循环
3. 难点：长上下文项目理解、代码修改的精确性（diff而非重写）、多文件依赖关系、测试覆盖
4. 优化：项目结构摘要、相关代码片段检索、渐进式修改、执行结果反馈

### 数据分析 Agent
**核心要点**：NL2SQL+可视化+洞察生成的数据智能分析
**常见面试问法**：
- "数据分析Agent怎么设计？"
- "NL2SQL准确率怎么提升？"
**回答框架**：
1. 架构：自然语言问题→NL2SQL生成查询→执行SQL→结果分析→可视化生成→洞察总结
2. 关键技术：NL2SQL（Schema感知的SQL生成）、代码执行（Python数据分析）、可视化（自动选择图表类型）、洞察生成（LLM解读数据模式）
3. 难点：复杂SQL生成准确率、Schema理解、多表关联、数据安全（只读查询+行级权限）
4. 优化：Few-shot示例库、Schema描述增强、SQL验证和纠错循环、查询结果缓存

### 工作流自动化 Agent
**核心要点**：多步骤任务编排+人工审核的企业流程自动化
**常见面试问法**：
- "工作流Agent怎么设计？"
- "人工审核节点怎么实现？"
**回答框架**：
1. 架构：任务输入→步骤拆解→逐步执行→关键节点人工审核→继续执行→输出结果
2. 关键技术：任务分解（LLM规划执行步骤）、工具编排（按顺序调用不同API/系统）、状态管理（记录每步执行状态）、人工审核（interrupt机制暂停等待人工确认）
3. 难点：步骤间依赖管理、异常处理和回滚、长时间运行的状态持久化、人工审核的超时处理
4. 优化：LangGraph的Checkpoint实现断点续跑、失败步骤重试、并行步骤加速、审核超时自动升级

## 十、项目高频追问

### Agent 可靠性 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：Agent可靠性是落地的最大挑战，需从架构、验证、兜底多层面保障
**常见面试问法**：
- "Agent输出不稳定怎么办？"
- "如何保证Agent可靠执行？"
- "Agent犯错怎么兜底？"
**回答框架**：
1. 不可靠来源：LLM生成随机性、工具调用失败、多步推理误差累积、上下文理解偏差
2. 提升可靠性：降低Temperature（0.0-0.1）、结构化输出（JSON Mode）、工具调用校验（参数类型/范围检查）、结果验证（LLM自检+规则校验）、重试机制（失败自动重试+换策略重试）
3. 兜底策略：关键操作人工确认、失败时降级到简单方案、输出不确定时标注置信度、异常时回滚到安全状态
4. 架构层面：幂等设计（重复调用不产生副作用）、事务性操作（多步操作可回滚）、超时熔断（防止无限循环）

### Agent 成本优化 [频率: ⭐⭐⭐⭐]
**核心要点**：Agent成本主要来自LLM调用，需从模型选择、调用次数、Prompt长度三方面优化
**常见面试问法**：
- "Agent成本太高怎么优化？"
- "怎么减少LLM调用次数？"
**回答框架**：
1. 成本构成：LLM API调用费用（占比最大，与token数和模型等级正相关）、向量数据库费用、基础设施费用
2. 模型选择：简单任务用小模型（GPT-4o-mini/Qwen-7B）、复杂任务用大模型、路由分类（先判断复杂度再选模型）
3. 调用次数优化：缓存相似查询结果、批量处理减少轮次、工具调用合并、Prompt压缩减少输入token
4. Prompt优化：精简系统提示词、历史对话摘要替代原文、只检索最相关片段、结构化输出减少冗余生成

### Agent 延迟优化 [频率: ⭐⭐⭐⭐]
**核心要点**：Agent延迟是多步串行累积的，需从并行化、流式输出、模型选择三方面优化
**常见面试问法**：
- "Agent响应太慢怎么优化？"
- "多步推理延迟怎么降低？"
**回答框架**：
1. 延迟来源：LLM推理延迟（主要）、工具调用延迟、检索延迟、多步串行累积
2. 并行化：无依赖的工具调用并行执行、多路检索并行、预取（预测下一步可能需要的数据提前加载）
3. 流式输出：LLM生成时流式返回中间结果、工具调用时先返回状态再返回结果、用户感知延迟大幅降低
4. 模型选择：简单判断用小模型（延迟低）、流式输出降低首token时间、推理框架优化（vLLM连续批处理）、模型蒸馏（大模型能力迁移到小模型）

### Agent 线上故障排查 [频率: ⭐⭐⭐]
**核心要点**：Agent故障排查依赖完善的Trace和日志，常见故障有超时、循环、幻觉
**常见面试问法**：
- "Agent线上出问题怎么排查？"
- "Agent死循环怎么发现和解决？"
**回答框架**：
1. 常见故障：超时（LLM响应慢或工具调用慢）、死循环（Agent反复调用同一工具或重复推理）、幻觉（输出与事实不符）、工具调用失败（参数错误或服务不可用）
2. 排查流程：查看Trace定位故障节点→检查该节点输入输出→分析是LLM问题还是工具问题→定位根因
3. 死循环检测：设置最大步数限制、检测重复工具调用模式、设置总超时时间
4. 预防：完善的监控告警（延迟/错误率/成本异常）、自动降级策略、故障复盘沉淀到测试用例

### Agent 与传统系统对比 [频率: ⭐⭐⭐⭐]
**核心要点**：Agent不是替代传统系统，而是在特定场景补充传统系统的不足
**常见面试问法**：
- "Agent能替代传统规则引擎吗？"
- "什么场景用Agent，什么场景用传统系统？"
- "Agent和传统系统怎么结合？"
**回答框架**：
1. Agent优势：处理非结构化输入、灵活适应新场景、自然语言交互、自主决策
2. 传统系统优势：确定性输出、高性能低延迟、成本低、可审计、易调试
3. 选型原则：确定性流程用传统系统、需要理解和决策的环节用Agent、两者结合取长补短
4. 结合模式：Agent做理解和规划层，传统系统做执行层；Agent处理异常case，传统系统处理标准case；Agent做意图路由，传统系统做具体业务逻辑

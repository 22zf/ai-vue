# 数据分析面试八股文库

## 一、SQL

### 窗口函数 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：窗口函数在不缩减行数的情况下进行分组计算，ROW_NUMBER/RANK/DENSE_RANK用于排名，LEAD/LAG用于前后行比较，是数据分析SQL最重要的考点。
**常见面试问法**：
- "ROW_NUMBER、RANK、DENSE_RANK有什么区别？"
- "怎么用SQL找出每个部门薪资前三的员工？"
**回答框架**：
1. 语法：函数名() OVER (PARTITION BY 分组列 ORDER BY 排序列 ROWS/RANGE 窗口范围)
2. 排名函数：ROW_NUMBER（连续不重复1,2,3）、RANK（并列跳号1,1,3）、DENSE_RANK（并列不跳1,1,2）
3. 偏移函数：LEAD(列,n)向后第n行、LAG(列,n)向前第n行，用于计算环比/同环
4. 聚合窗口：SUM() OVER()累计求和、AVG() OVER()移动平均
**延伸考点**：JOIN → 留存率计算

### JOIN连接查询 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：JOIN是SQL最核心的多表查询能力，INNER JOIN取交集、LEFT JOIN保留左表、FULL JOIN保留全部，理解连接逻辑是写对SQL的前提。
**常见面试问法**：
- "LEFT JOIN和INNER JOIN有什么区别？"
- "LEFT JOIN后右表有重复数据怎么办？"
**回答框架**：
1. INNER JOIN：只返回两表匹配的行，不匹配的丢弃
2. LEFT JOIN：返回左表全部行，右表无匹配则填NULL
3. RIGHT JOIN：返回右表全部行，左表无匹配则填NULL
4. FULL JOIN：返回两表全部行，无匹配填NULL
5. 注意：LEFT JOIN右表有重复会导致左表行数膨胀，需先去重或用子查询
**延伸考点**：子查询 → 窗口函数

### 聚合与分组 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：GROUP BY分组+聚合函数实现分组统计，HAVING过滤分组结果，执行顺序FROM→WHERE→GROUP BY→HAVING→SELECT→ORDER BY。
**常见面试问法**：
- "WHERE和HAVING有什么区别？"
- "GROUP BY后SELECT能写什么？"
**回答框架**：
1. GROUP BY：按指定列分组，每组返回一行聚合结果
2. 聚合函数：COUNT/SUM/AVG/MAX/MIN，COUNT(*)含NULL，COUNT(列)不含NULL
3. HAVING vs WHERE：WHERE过滤原始行（GROUP BY前），HAVING过滤分组结果（GROUP BY后）
4. SELECT规则：GROUP BY后SELECT只能出现分组列和聚合函数，否则报错（MySQL默认宽松模式可能不报但结果不确定）
**延伸考点**：窗口函数 → 漏斗分析

### 找第N高的薪水 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：找第N高是SQL经典题型，核心方法有LIMIT OFFSET、DENSE_RANK窗口函数、自连接三种，需处理重复值和NULL情况。
**常见面试问法**：
- "怎么找出第二高的薪水？如果没有返回NULL？"
- "如果有重复薪水怎么处理？"
**回答框架**：
1. LIMIT OFFSET：SELECT DISTINCT salary FROM employee ORDER BY salary DESC LIMIT 1 OFFSET 1
2. DENSE_RANK：SELECT salary FROM (SELECT salary, DENSE_RANK() OVER(ORDER BY salary DESC) rnk FROM employee) t WHERE rnk = 2
3. IFNULL处理：IFNULL((SELECT DISTINCT salary ... LIMIT 1 OFFSET 1), NULL)
4. 重复值处理：必须用DISTINCT或DENSE_RANK，不能用ROW_NUMBER（会跳号）
**延伸考点**：窗口函数 → 连续登录问题

### 连续登录问题 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：连续登录是SQL高频难题，核心思路是用ROW_NUMBER减去日期排序得到分组标识，相同标识的行即为连续。
**常见面试问法**：
- "怎么找出连续3天登录的用户？"
- "连续登录问题的通用解法是什么？"
**回答框架**：
1. 核心思路：日期 - ROW_NUMBER() = 分组标识，连续日期的差值相同
2. 步骤：按用户分组→按日期排序生成ROW_NUMBER→用日期减去序号得到分组ID→按分组ID计数≥3
3. SQL：SELECT user_id FROM (SELECT user_id, login_date, DATE_SUB(login_date, INTERVAL rn DAY) AS grp FROM (SELECT user_id, login_date, ROW_NUMBER() OVER(PARTITION BY user_id ORDER BY login_date) AS rn FROM logins) t) t2 GROUP BY user_id, grp HAVING COUNT(*) >= 3
4. 变体：连续N天、最大连续天数、断一天也算连续（用DATE_SUB减去rn*2）
**延伸考点**：窗口函数 → 留存率计算

### 留存率计算 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：留存率 = 第N天回访用户数 / 首日新增用户数，核心是找到每个用户的首次登录日和后续登录日的差值。
**常见面试问法**：
- "怎么计算次日留存率、7日留存率？"
- "留存率的SQL怎么写？"
**回答框架**：
1. 定义：次日留存 = 次日回访用户数 / 首日新增用户数 × 100%
2. 步骤：找到每个用户首次登录日期→计算后续登录与首次登录的天数差→按天数差统计人数
3. SQL核心：FIRST_VALUE(login_date) OVER(PARTITION BY user_id ORDER BY login_date)获取首次日期，DATEDIFF计算差值
4. 同期群分析：按首次登录日期分组，计算每个同期群的N日留存率
**延伸考点**：连续登录 → 同期群分析

### 用户漏斗分析 [频率: ⭐⭐⭐⭐]
**核心要点**：漏斗分析统计用户在各步骤的转化率，核心是找到每个用户在每一步的最早时间，然后判断是否按顺序完成。
**常见面试问法**：
- "怎么用SQL做漏斗分析？"
- "漏斗中每一步的转化率怎么算？"
**回答框架**：
1. 定义：漏斗每一步人数 / 第一步人数 = 该步转化率
2. 步骤：找到每个用户在每个步骤的最早时间→判断步骤间时间顺序是否正确→统计每步人数
3. SQL：用CASE WHEN+MIN聚合每个步骤的时间，WHERE条件确保步骤时间递增
4. 进阶：窗口函数实现有序漏斗、时间窗口限制（如30分钟内完成）
**延伸考点**：留存率 → 归因分析

### 同比环比计算 [频率: ⭐⭐⭐⭐]
**核心要点**：环比是与上一周期比较（月环比=本月/上月-1），同比是与去年同期比较（月同比=本月/去年同月-1），用LAG窗口函数实现。
**常见面试问法**：
- "同比和环比有什么区别？"
- "怎么用SQL计算同比环比？"
**回答框架**：
1. 环比：当前值与上一周期比较，LAG(指标, 1) OVER(ORDER BY 日期)
2. 同比：当前值与去年同期比较，LAG(指标, 12) OVER(ORDER BY 月份)（月度数据）
3. 计算公式：环比增长率 = (当前值 - 上期值) / 上期值 × 100%
4. 注意：同比需处理闰年2月、节假日偏移等特殊情况
**延伸考点**：窗口函数 → 指标体系设计

## 二、Python数据分析

### Pandas核心操作 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：Pandas以DataFrame为核心数据结构，支持数据读取、清洗、变换、聚合、合并全流程，是Python数据分析的基石。
**常见面试问法**：
- "Pandas的DataFrame和Series有什么区别？"
- "groupby+agg怎么用？"
**回答框架**：
1. 数据结构：Series一维带标签数组、DataFrame二维表格，Index行标签、Columns列标签
2. 数据读取：pd.read_csv/read_excel/read_sql，指定encoding/dtype/parse_dates
3. 分组聚合：df.groupby('列').agg({'列1': 'mean', '列2': 'sum'})，支持自定义聚合函数
4. 合并：pd.merge()类似SQL JOIN、pd.concat()纵向/横向拼接、df.join()按索引合并
**延伸考点**：数据清洗 → NumPy广播

### NumPy广播机制 [频率: ⭐⭐⭐⭐]
**核心要点**：广播是NumPy对不同形状数组进行算术运算的机制，小数组自动"扩展"到大数组的形状，避免显式循环提升性能。
**常见面试问法**：
- "什么是广播？广播的规则是什么？"
- "什么情况下两个数组不能广播？"
**回答框架**：
1. 规则：从最右维度开始比较，维度相同或其中一个为1则兼容，否则报错
2. 示例：(3,4) + (4,) → (4,)扩展为(3,4)；(3,1) + (1,4) → 双方扩展为(3,4)
3. 不兼容：(3,4) + (3,) → 最右维度4≠3且都不为1，报错
4. 优势：避免Python循环，利用C层向量化运算，性能提升数十倍
**延伸考点**：Pandas操作 → 数据清洗

### 数据清洗 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：数据清洗占数据分析60%以上工作量，核心处理缺失值、异常值、重复值，清洗质量直接决定分析结论可靠性。
**常见面试问法**：
- "缺失值怎么处理？"
- "怎么检测和处理异常值？"
**回答框架**：
1. 缺失值：df.isnull().sum()统计→df.fillna()填充（均值/中位数/前向填充）→df.dropna()删除
2. 异常值：3σ原则（超出3倍标准差）、箱线图（IQR×1.5）、业务规则判断→删除/截断/替换
3. 重复值：df.duplicated()检测→df.drop_duplicates()删除
4. 数据类型：df.astype()转换类型、pd.to_datetime()转换日期、str方法清洗字符串
**延伸考点**：Pandas操作 → 业务分析

### 数据可视化 [频率: ⭐⭐⭐]
**核心要点**：Matplotlib是基础绑图库灵活但繁琐，Seaborn基于Matplotlib封装统计图表更简洁，选择取决于需求复杂度。
**常见面试问法**：
- "Matplotlib和Seaborn有什么区别？"
- "怎么选择合适的图表类型？"
**回答框架**：
1. Matplotlib：底层绑图库，灵活可控，适合定制化图表，但代码量大
2. Seaborn：统计可视化库，一行代码出图，内置统计图表（分布图/热力图/箱线图）
3. 图表选型：趋势用折线图、比较用柱状图、占比用饼图/环形图、分布用直方图/箱线图、关系用散点图/热力图
4. 实践：探索阶段用Seaborn快速出图，报告阶段用Matplotlib精细调整
**延伸考点**：数据清洗 → 业务分析报告

## 三、统计学

### 描述统计 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：描述统计用集中趋势（均值/中位数/众数）和离散程度（方差/标准差/四分位距）概括数据特征，是数据分析的第一步。
**常见面试问法**：
- "均值和中位数有什么区别？什么时候用中位数？"
- "标准差和四分位距有什么区别？"
**回答框架**：
1. 集中趋势：均值（受极端值影响大）、中位数（抗极端值）、众数（分类数据）
2. 离散程度：方差/标准差（受极端值影响）、四分位距IQR=Q3-Q1（抗极端值）
3. 选用：数据对称用均值+标准差，数据偏态用中位数+IQR
4. 分布形态：偏度（左偏/右偏）、峰度（尖峰/平峰），判断数据是否正态
**延伸考点**：概率分布 → 假设检验

### 概率分布 [频率: ⭐⭐⭐⭐]
**核心要点**：正态分布是统计推断的基础，二项分布描述离散事件次数，泊松分布描述稀有事件频率，三者是数据分析最常见的分布。
**常见面试问法**：
- "正态分布有什么性质？"
- "二项分布和泊松分布有什么区别？"
**回答框架**：
1. 正态分布：钟形对称、均值=中位数=众数、68-95-99.7法则、中心极限定理保证样本均值趋近正态
2. 二项分布：n次独立试验中成功k次的概率，参数n和p，如抛硬币n次正面朝上次数
3. 泊松分布：单位时间/空间内稀有事件发生次数，参数λ（平均发生率），如每小时客服来电数
4. 关系：n大p小时二项分布近似泊松分布，n大时二项分布近似正态分布
**延伸考点**：描述统计 → 假设检验

### 假设检验 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：假设检验通过样本数据判断总体参数是否成立，p值<α则拒绝原假设，t检验比较均值、卡方检验比较比例/独立性。
**常见面试问法**：
- "p值是什么意思？0.05的显著性水平怎么理解？"
- "t检验和卡方检验分别适用于什么场景？"
**回答框架**：
1. 基本流程：建立原假设H0和备择假设H1→选择检验方法→计算统计量和p值→p<α则拒绝H0
2. p值：在H0为真的前提下，观察到当前或更极端结果的概率，p<0.05表示结果不太可能由随机产生
3. t检验：比较两组均值差异（独立样本t检验/配对t检验），要求数据近似正态
4. 卡方检验：比较分类变量的比例差异或独立性，适用于频数数据
**延伸考点**：A/B测试 → 置信区间

### 置信区间 [频率: ⭐⭐⭐⭐]
**核心要点**：置信区间给出总体参数的估计范围，95%置信区间表示重复抽样100次约95次包含真实参数，区间越窄估计越精确。
**常见面试问法**：
- "95%置信区间是什么意思？"
- "置信区间宽度受什么影响？"
**回答框架**：
1. 定义：总体参数以一定概率落在的区间范围，95%置信区间 = 点估计 ± 1.96 × 标准误
2. 正确理解：不是"参数有95%概率落在此区间"，而是"此方法构造的区间有95%包含真实参数"
3. 影响因素：样本量↑区间↓（更精确）、置信水平↑区间↑（更保守）、数据方差↑区间↑
4. 与假设检验关系：若H0值不在置信区间内，则拒绝H0，两者等价
**延伸考点**：假设检验 → A/B测试

### 相关性 vs 因果性 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：相关性表示变量间的统计关联，因果性表示一个变量导致另一个变量变化，相关不等于因果，因果推断需要实验设计。
**常见面试问法**：
- "相关性和因果性有什么区别？"
- "怎么证明因果关系？"
**回答框架**：
1. 相关性：变量间存在统计关联，用相关系数衡量（-1到1），可能由混淆变量导致
2. 因果性：X的变化导致Y的变化，需满足：时序性（X先于Y）、关联性、排除混淆变量
3. 混淆变量：冰淇淋销量和溺水人数正相关，混淆变量是温度，非因果关系
4. 因果推断：随机对照实验（A/B测试）是金标准，观察数据可用DID/IV/PSM等方法
**延伸考点**：假设检验 → A/B测试

### A/B测试 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：A/B测试通过随机对照实验验证策略效果，核心是样本量计算、随机分流、显著性检验和效果评估。
**常见面试问法**：
- "A/B测试怎么设计的？样本量怎么算？"
- "A/B测试结果不显著怎么办？"
**回答框架**：
1. 设计流程：确定指标→计算样本量→随机分流→运行实验→显著性检验→决策
2. 样本量：n = (Z_α/2 + Z_β)² × 2p(1-p) / δ²，δ为最小可检测效应，p为基准转化率
3. 检验方法：连续指标用t检验、转化率用比例检验（Z检验）、分类用卡方检验
4. 不显著处理：检查样本量是否足够、实验周期是否覆盖周期性、分流是否均匀、指标是否合理
**延伸考点**：假设检验 → 置信区间

## 四、业务分析

### 指标体系设计 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：指标体系从北极星指标→一级指标→二级指标逐层拆解，遵循OSM模型（目标-策略-度量），确保指标可衡量可归因。
**常见面试问法**：
- "怎么设计指标体系？"
- "DAU下降了怎么分析？"
**回答框架**：
1. 北极星指标：衡量产品核心价值的单一指标（如GMV/日活/使用时长）
2. 指标拆解：北极星→一级指标（获客/活跃/留存/变现）→二级指标（渠道/功能/页面级）
3. OSM模型：Objective（业务目标）→Strategy（实现策略）→Measurement（度量指标）
4. DAU下降分析：外部因素（节假日/竞品/政策）→内部因素（技术故障/版本更新/运营活动结束）→拆解维度（渠道/地区/新老用户）
**延伸考点**：漏斗分析 → 异常指标排查

### 漏斗分析 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：漏斗分析衡量用户在业务流程各步骤的转化率，核心是定义步骤、统计人数、计算转化率、定位流失环节。
**常见面试问法**：
- "漏斗分析怎么做？"
- "转化率低怎么优化？"
**回答框架**：
1. 定义步骤：明确业务流程的关键节点（如：浏览→加购→下单→支付）
2. 计算转化：每步人数 / 第一步人数 = 整体转化率，相邻步人数比 = 步骤转化率
3. 定位流失：找到转化率最低的步骤，分析流失用户特征（新/老、渠道、设备）
4. 优化策略：缩短步骤、优化UI/UX、增加引导/激励、A/B测试验证
**延伸考点**：指标体系 → 同期群分析

### 同期群分析（Cohort） [频率: ⭐⭐⭐⭐⭐]
**核心要点**：同期群分析按用户首次出现时间分组，追踪每组的后续行为指标，用于评估产品改进对用户留存的长期影响。
**常见面试问法**：
- "什么是同期群分析？有什么用？"
- "怎么用SQL做同期群分析？"
**回答框架**：
1. 定义：按首次出现时间（注册日/首次购买日）将用户分组，追踪每组在后续时间窗口的指标
2. 价值：区分新老用户表现、评估产品改版效果、发现留存趋势变化
3. SQL实现：首次日期+DATEDIFF计算天数差→按首次日期和天数差分组→计算每组的留存率
4. 可视化：热力图展示，横轴天数差、纵轴同期群、颜色深浅表示留存率
**延伸考点**：留存率计算 → 归因分析

### 归因分析 [频率: ⭐⭐⭐⭐]
**核心要点**：归因分析确定各触点对转化的贡献权重，首次触点归因归功于首次接触、末次触点归因归功于最后接触、线性归因均分权重。
**常见面试问法**：
- "归因分析有哪些模型？"
- "末次触点归因有什么问题？"
**回答框架**：
1. 首次触点归因：100%归功于用户首次接触的渠道，适合品牌认知阶段
2. 末次触点归因：100%归功于转化前的最后一个渠道，适合效果广告
3. 线性归因：所有触点均分权重，简单但忽略触点重要性差异
4. 数据驱动归因：基于统计模型（马尔可夫链/Shapley值）计算各触点贡献，最科学但需大量数据
**延伸考点**：同期群分析 → 指标体系

### 异常指标排查 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：异常指标排查遵循"先确认异常→再拆解定位→最后归因"的流程，核心是维度拆解和对比分析。
**常见面试问法**：
- "指标下降了怎么排查？"
- "怎么快速定位异常原因？"
**回答框架**：
1. 确认异常：是否真的异常（同比/环比/历史波动范围）、数据是否准确（埋点/ETL是否正常）
2. 维度拆解：按渠道/地区/设备/新老用户/版本逐层拆解，找到异常集中在哪个维度
3. 内因排查：技术故障、版本发布、运营活动结束、策略调整
4. 外因排查：节假日、竞品动作、政策变化、行业趋势
5. 验证假设：用数据验证每个可能原因，排除法定位根因
**延伸考点**：指标体系 → 漏斗分析

## 五、数据仓库

### 数仓分层 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：数仓分层ODS→DWD→DWS→ADS，从原始数据到应用数据逐层加工，每层职责清晰，保证数据质量和复用性。
**常见面试问法**：
- "数仓为什么要分层？每层做什么？"
- "DWD和DWS有什么区别？"
**回答框架**：
1. ODS（操作数据层）：原始数据1:1同步，不加工，保留历史全量
2. DWD（明细数据层）：清洗+标准化+维度退化，一行一条业务事件，粒度最细
3. DWS（汇总数据层）：按主题+时间周期轻度汇总，如用户日汇总表、商品日汇总表
4. ADS（应用数据层）：面向具体业务场景的宽表/指标表，直接供报表/分析使用
5. 价值：复用中间层减少重复计算、统一口径保证一致性、问题逐层排查
**延伸考点**：ETL流程 → 维度建模

### ETL流程 [频率: ⭐⭐⭐⭐]
**核心要点**：ETL是数据抽取→转换→加载的过程，是数仓建设的核心环节，数据质量取决于ETL的健壮性。
**常见面试问法**：
- "ETL流程怎么设计的？"
- "数据质量怎么保证？"
**回答框架**：
1. 抽取（Extract）：全量抽取（首次/小表）、增量抽取（CDC/时间戳/日志解析）
2. 转换（Transform）：数据清洗（去重/空值/异常值）、格式统一、编码转换、业务逻辑计算
3. 加载（Load）：全量覆盖、增量追加、upsert合并
4. 数据质量：空值检查、唯一性检查、范围检查、一致性检查、监控告警
**延伸考点**：数仓分层 → 数据质量监控

### 维度建模 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：维度建模以业务过程为中心，星型模型事实表居中维度表环绕（允许冗余），雪花模型维度表进一步规范化（减少冗余）。
**常见面试问法**：
- "星型模型和雪花模型有什么区别？"
- "事实表和维度表有什么区别？"
**回答框架**：
1. 事实表：记录业务事件，包含度量值（可聚合的数值）和外键（关联维度），行数多、增长快
2. 维度表：描述业务事件的上下文（谁/何时/何地/什么），行数少、变化慢
3. 星型模型：维度表直接关联事实表，维度表允许冗余（如省市区冗余在地区维度），查询性能好
4. 雪花模型：维度表进一步规范化拆分（省/市/区分表），减少冗余但JOIN多、查询慢
5. 实践：大多数场景选星型模型，查询性能优先
**延伸考点**：数仓分层 → Hive SQL

### Hive SQL [频率: ⭐⭐⭐⭐]
**核心要点**：Hive是基于HDFS的数据仓库工具，用HQL（类SQL）查询大数据，核心概念是分区/分桶/文件格式，理解执行计划是优化的关键。
**常见面试问法**：
- "Hive的分区和分桶有什么区别？"
- "Hive SQL怎么优化？"
**回答框架**：
1. 分区：按分区列将数据存储在不同目录，查询时分区裁剪只读相关目录，减少IO
2. 分桶：按分桶列哈希将数据分散到固定数量的文件，用于优化JOIN和数据采样
3. 文件格式：TextFile（默认）、ORC（列存+压缩，推荐）、Parquet（列存+嵌套支持）
4. 优化：分区裁剪、MapJoin（小表放内存）、数据倾斜处理（加盐/分桶）、合理设置Reducer数量
**延伸考点**：维度建模 → 数据倾斜处理

### 数据质量监控 [频率: ⭐⭐⭐⭐]
**核心要点**：数据质量监控从完整性/准确性/一致性/及时性四个维度保障数据可靠，是数仓运营的核心能力。
**常见面试问法**：
- "数据质量怎么保障？"
- "发现数据异常怎么处理？"
**回答框架**：
1. 完整性：行数校验（与源系统对比）、空值率监控、主键唯一性检查
2. 准确性：业务规则校验（金额>0、日期合理）、与源系统对账、抽样人工校验
3. 一致性：跨表口径一致、跨层指标一致、同一指标不同报表结果一致
4. 及时性：任务完成时间监控、SLA告警、关键路径任务优先级保障
5. 异常处理：告警→排查→修复→重跑→复盘→加监控规则
**延伸考点**：ETL流程 → 数仓分层

## 六、高频项目方向

### 用户行为分析 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：基于埋点数据构建用户画像和行为路径，核心是埋点设计→数据采集→行为分析→画像构建→策略输出。
**常见面试问法**：
- "用户行为分析怎么做的？"
- "埋点怎么设计的？"
**回答框架**：
1. 埋点设计：事件埋点（点击/曝光/页面停留）+属性（用户/设备/页面/时间），遵循统一命名规范
2. 数据采集：前端埋点（代码埋点/可视化埋点/全埋点）→数据上报→落盘ODS
3. 行为分析：路径分析（用户行为序列）、漏斗分析（转化率）、留存分析（回访率）
4. 用户画像：基础属性+行为标签+RFM分层+兴趣偏好，输出分群策略
**延伸考点**：指标体系 → 漏斗分析

### A/B测试分析项目 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：A/B测试分析项目覆盖实验设计→指标定义→样本量计算→效果评估→决策建议全流程，体现统计+业务的综合能力。
**常见面试问法**：
- "A/B测试分析怎么做的？"
- "实验结果怎么评估的？"
**回答框架**：
1. 实验设计：确定假设→选择指标（核心指标+护栏指标）→计算样本量→确定实验周期
2. 分流验证：SRM检验（样本比例偏差检测）、分流均匀性验证、特征分布一致性
3. 效果评估：核心指标提升+显著性检验、护栏指标未恶化、异质性分析（分群体看效果差异）
4. 决策建议：效果显著→全量上线；不显著→分析原因（样本量/周期/策略）→迭代实验
**延伸考点**：A/B测试 → 假设检验

### 指标监控Dashboard [频率: ⭐⭐⭐⭐]
**核心要点**：指标监控Dashboard实现关键业务指标的实时/准实时监控，核心是指标定义→数据链路→可视化→告警机制。
**常见面试问法**：
- "指标监控怎么做的？"
- "告警策略怎么设计的？"
**回答框架**：
1. 指标定义：核心指标+拆解维度，统一口径和计算逻辑
2. 数据链路：埋点→Kafka→Flink/Spark实时计算→OLAP引擎（ClickHouse/Doris）→可视化
3. 可视化：Superset/Metabase/自研看板，支持多维下钻和对比分析
4. 告警：阈值告警（同比/环比超阈值）、智能告警（基于历史波动3σ）、告警分级和通知渠道
**延伸考点**：异常指标排查 → 数据质量监控

## 七、项目高频追问

### 指标下降排查追问 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：这是数据分析面试最高频的追问，考察系统性思维和业务敏感度，关键是结构化拆解而非猜测。
**常见面试问法**：
- "指标下降了怎么排查？"
- "DAU下降了20%，你怎么分析？"
**回答框架**：
1. 确认数据：先确认数据是否准确（埋点/ETL/统计口径是否有变更）
2. 定位范围：时间范围（何时开始下降）、空间范围（哪个渠道/地区/版本）
3. 维度拆解：新老用户→渠道→地区→设备→版本，找到异常集中的维度
4. 内因排查：技术故障、版本发布、运营活动、策略调整
5. 外因排查：节假日、竞品、政策、行业趋势
6. 验证假设：用数据验证每个假设，排除法定位根因
**延伸考点**：异常指标排查 → 指标体系

### A/B测试样本量追问 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：追问考察你对统计原理的理解，而非只会套公式，重点是理解样本量与效应量、显著性水平、统计功效的关系。
**常见面试问法**：
- "A/B测试样本量怎么算的？"
- "样本量不够怎么办？"
**回答框架**：
1. 公式：n = (Z_α/2 + Z_β)² × 2p(1-p) / δ²，α=0.05, β=0.2, p=基准转化率, δ=最小可检测效应
2. 参数影响：δ越小样本量越大（检测更小的差异需要更多样本）、α越小样本量越大、β越小样本量越大
3. 样本量不够：延长实验周期、扩大分流比例、降低δ（接受更大差异才显著）、合并相似实验组
4. 实际考虑：需考虑周期间效应（day-of-week）、新奇效应、溢出效应
**延伸考点**：A/B测试 → 假设检验

### 数据清洗追问 [频率: ⭐⭐⭐⭐]
**核心要点**：追问考察你对数据质量问题的敏感度和处理经验，而非仅会调用API。
**常见面试问法**：
- "数据来源是什么？怎么清洗的？"
- "遇到脏数据怎么处理？"
**回答框架**：
1. 数据来源：埋点数据（前端/后端）、业务数据库（MySQL同步）、第三方数据（API/爬虫）
2. 清洗流程：去重→缺失值处理→异常值处理→格式统一→口径对齐
3. 脏数据案例：埋点重复上报（幂等去重）、时区不统一（统一UTC）、金额单位不一致（统一分/元）
4. 质量保障：清洗规则文档化、自动化质量检查、异常数据告警、定期数据对账
**延伸考点**：数据清洗 → ETL流程

### 分析结论与决策推动追问 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：追问考察你的分析是否真正影响了业务决策，而非仅产出报告，体现数据分析的核心价值。
**常见面试问法**：
- "分析结论是什么？推动了什么决策？"
- "分析结果和业务预期不一致怎么办？"
**回答框架**：
1. 结论呈现：用数据说话，核心发现+数据支撑+可视化，结论简洁明确
2. 决策推动：提出可执行建议（而非仅描述问题）、量化预期收益、与业务方对齐优先级
3. 不一致处理：先检查数据/方法是否正确→再与业务方讨论假设差异→可能发现新的业务洞察
4. 闭环验证：建议落地后追踪效果，形成"分析→决策→验证"闭环
**延伸考点**：指标体系 → 业务分析

### SQL优化追问 [频率: ⭐⭐⭐⭐]
**核心要点**：SQL优化是数据分析师的硬技能，追问考察你对执行原理的理解和实际优化经验。
**常见面试问法**：
- "SQL优化有什么经验？"
- "慢查询怎么优化？"
**回答框架**：
1. 索引优化：WHERE/JOIN/ORDER BY列建索引、联合索引遵循最左前缀、避免索引失效
2. 查询优化：避免SELECT *、用EXISTS替代IN（大数据集）、用JOIN替代子查询、LIMIT限制结果
3. 数据倾斜：大表JOIN小表用MapJoin、倾斜Key单独处理（加盐打散）、分桶优化
4. 执行计划：EXPLAIN查看执行计划→定位全表扫描/笛卡尔积/数据倾斜→针对性优化
**延伸考点**：Hive SQL → 数据倾斜处理

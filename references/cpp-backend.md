# C++ 后端面试八股文库

## 一、语言基础

### STL容器 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：STL容器是C++标准库的核心组件，理解各容器底层实现（连续内存vs节点存储）是选型和回答性能问题的关键。

**常见面试问法**：
- "vector的扩容机制是怎样的？为什么选择1.5倍或2倍？"
- "哪些操作会导致vector/map/unordered_map的迭代器失效？"

**回答框架**：
1. 先按底层结构分类：连续内存型（vector、deque）和节点型（list、map、set、unordered_map）
2. 逐个说明扩容/重组机制：vector 2倍扩容触发整体搬迁，unordered_map 负载因子超阈值rehash
3. 迭代器失效规则：vector插入可能全部失效，删除使被删及之后失效；map删除仅被删节点失效；unordered_map扩容全部失效
4. 给出容器选型建议：频繁随机访问选vector，频繁中间插入选list，有序查找选map，无序高频查找选unordered_map

**延伸考点**：迭代器失效 → reserve/shrink_to_fit优化 → 容器线程安全性

### vector扩容机制 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：vector在容量不足时以2倍（GCC）或1.5倍（MSVC）扩容，开辟新空间、拷贝/移动元素、释放旧空间，均摊时间复杂度O(1)。

**常见面试问法**：
- "vector push_back的时间复杂度是多少？为什么？"
- "为什么扩容因子选1.5或2？1行不行？"

**回答框架**：
1. 说明扩容触发条件：size == capacity时触发
2. 扩容过程：申请新空间 → 拷贝/移动旧元素 → 释放旧空间 → 更新指针
3. 均摊分析：n次push_back总拷贝次数为1+2+4+...+n ≈ 2n，均摊O(1)
4. 扩容因子选择：2倍可能导致内存永远无法复用（每次申请都大于之前释放之和），1.5倍可以让释放的内存块组合复用；1倍则退化为每次都扩容，效率极差
5. 优化手段：reserve预分配、emplace_back原地构造避免临时对象

**延伸考点**：push_back vs emplace_back → move语义在扩容中的应用 → small object optimization

### 迭代器失效场景 [频率: ⭐⭐⭐⭐]

**核心要点**：迭代器失效的根本原因是底层内存发生了重新分配或节点被删除，不同容器因底层结构不同，失效规则也不同。

**常见面试问法**：
- "删除vector中所有偶数元素，怎么写才安全？"
- "map删除元素后迭代器一定失效吗？"

**回答框架**：
1. vector：插入可能全部失效（扩容），删除使被删位置及之后失效；安全删除用 `it = vec.erase(it)`
2. deque：中间插入/删除全部失效，首尾插入仅迭代器失效引用仍有效
3. list/map/set：删除仅使被删节点迭代器失效，其他不受影响；安全删除用 `it++` 或 `it = map.erase(it)`
4. unordered_map：插入可能触发rehash导致全部失效，删除仅被删节点失效
5. 总结规律：连续内存容器失效范围大，节点容器失效范围小

**延伸考点**：erase-remove idiom → 反向迭代器失效规则 → C++11 erase返回值优化

### 智能指针 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：智能指针通过RAII管理堆资源生命周期，shared_ptr用引用计数共享所有权，unique_ptr独占所有权，weak_ptr解决循环引用。

**常见面试问法**：
- "shared_ptr的循环引用怎么解决？"
- "make_shared比new创建shared_ptr好在哪里？"

**回答框架**：
1. 三种智能指针定位：shared_ptr共享所有权、unique_ptr独占所有权、weak_ptr观察者不参与所有权
2. shared_ptr原理：控制块存储强引用计数和弱引用计数，计数为0时销毁对象/控制块
3. 循环引用场景：A持有B的shared_ptr，B持有A的shared_ptr，计数永不为0；解决方案是将一方改为weak_ptr
4. unique_ptr特性：不可拷贝但可移动，零开销抽象，适合工厂返回值和Pimpl惯用法
5. make_shared优势：一次分配同时分配对象和控制块（缓存友好）、异常安全（防止函数参数求值顺序导致的泄漏）
6. make_shared劣势：对象和控制块一起分配，强引用为0后若弱引用仍存在则对象内存无法释放

**延伸考点**：enable_shared_from_this → 自定义删除器 → shared_ptr线程安全性（计数原子但对象访问非线程安全）

### shared_ptr循环引用 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：两个对象互相持有对方的shared_ptr，导致引用计数永不为0，内存泄漏；用weak_ptr打破循环。

**常见面试问法**：
- "写一个shared_ptr循环引用的例子，并说明如何修复"
- "weak_ptr的lock()方法返回什么？如果对象已销毁呢？"

**回答框架**：
1. 给出经典场景：链表节点Node的next和prev都是shared_ptr，双向链表无法自动释放
2. 分析原因：A->B使B计数+1，B->A使A计数+1，离开作用域后各计数减为1而非0
3. 解决方案：将prev改为weak_ptr，不增加引用计数，A离开作用域后计数归0正常释放
4. weak_ptr使用：通过 `lock()` 获取shared_ptr（对象存活时返回有效shared_ptr，已销毁返回空），通过 `expired()` 检查是否过期
5. 补充：观察者模式中也常用weak_ptr避免观察者被强制延长生命周期

**延伸考点**：weak_ptr → enable_shared_from_this → shared_from_this()在构造函数中调用的陷阱

### unique_ptr与移动语义 [频率: ⭐⭐⭐⭐]

**核心要点**：unique_ptr通过删除拷贝构造、仅提供移动构造来实现独占所有权，转移所有权时必须用std::move。

**常见面试问法**：
- "unique_ptr可以拷贝吗？怎么转移所有权？"
- "函数返回unique_ptr需要用std::move吗？"

**回答框架**：
1. unique_ptr禁止拷贝：拷贝构造和拷贝赋值 = delete
2. 移动构造：转移内部指针和控制权，源unique_ptr变为nullptr
3. 所有权转移：`std::move(up)` 将左值转为右值引用触发移动构造
4. 函数返回值：返回局部unique_ptr时编译器执行RVO或隐式移动，不需要显式std::move（写了反而阻止RVO）
5. 典型应用：工厂模式返回unique_ptr、Pimpl惯用法、自定义删除器管理非内存资源（如文件句柄、互斥锁）

**延伸考点**：unique_ptr → 移动语义 → RVO/NRVO → C++14 make_unique

### C++11/14/17/20新特性 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：现代C++通过移动语义减少拷贝、通过类型推导简化代码、通过constexpr推向编译期计算、通过concepts约束模板。

**常见面试问法**：
- "右值引用和移动语义解决了什么问题？"
- "完美转发的原理是什么？为什么需要forward？"

**回答框架**：
1. C++11核心：右值引用（&&）绑定到临时对象，移动构造/移动赋值"窃取"资源而非深拷贝
2. 完美转发：模板中 `T&&` 是转发引用（universal reference），配合 `std::forward<T>` 保持值类别（左值/右值）不变地传递
3. lambda：捕获方式（值捕获、引用捕获、初始化捕获C++14）、泛型lambda（C++14）
4. C++14：返回类型推导、`decltype(auto)`、`make_unique`
5. C++17：结构化绑定、`optional`/`variant`/`any`、`if constexpr`、`std::filesystem`、折叠表达式
6. C++20：concepts（约束模板参数）、ranges、coroutines、`std::format`、三路比较运算符 `<=>`

**延伸考点**：移动语义 → 返回值优化RVO → 零拷贝 → concepts替代SFINAE

### 移动语义与完美转发 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：移动语义通过"窃取"资源避免深拷贝，完美转发保持参数的值类别（左值/右值）和const属性不变地传递给下游。

**常见面试问法**：
- "移动构造函数什么时候被调用？移动后原对象还能用吗？"
- "std::forward和std::move的区别？"

**回答框架**：
1. 左值与右值：左值有持久身份可取地址，右值是临时对象不可取地址
2. 移动语义：移动构造 `T(T&& other)` 将other的资源指针直接转移，other置为安全状态（如nullptr），而非重新分配内存
3. 移动后原对象：处于"有效但未指定"状态，只能赋值或析构，不能直接使用其值
4. 完美转发问题：右值传入模板参数后变成命名变量（左值），需要forward还原
5. `std::forward<T>(arg)`：T为左值引用时返回左值，T为非引用时返回右值；`std::move` 无条件转为右值
6. 转发引用（universal reference）：`template<typename T> void f(T&& arg)` 中T&&根据实参推导为左值引用或右值引用

**延伸考点**：引用折叠规则 → RVO与移动语义的关系 → vector扩容时的移动语义

### 面向对象与虚函数 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：C++通过虚函数表（vtable）和虚函数指针（vptr）实现运行时多态，每个含虚函数的类有虚表，每个对象有虚指针指向所属类的虚表。

**常见面试问法**：
- "虚函数的调用机制是怎样的？有什么性能开销？"
- "构造函数可以是虚函数吗？为什么？"

**回答框架**：
1. 虚表机制：编译器为每个含虚函数的类生成vtable，存储虚函数指针；对象构造时vptr指向对应类的vtable
2. 多态调用：通过基类指针调用虚函数时，通过vptr查vtable确定实际调用的函数，即动态绑定
3. 性能开销：每个对象多一个vptr（8字节）、虚表查找一次间接跳转、阻碍内联和编译期优化
4. 构造函数不能为虚函数：构造时对象类型尚未完整，vptr还未指向正确虚表
5. 析构函数应为虚函数：基类指针指向派生类对象时，若析构非虚则只调用基类析构，导致派生类资源泄漏
6. 构造/析构中调虚函数：调用的是当前类（而非派生类）的版本，因为派生类部分尚未构造/已析构

**延伸考点**：虚继承的虚表布局 → 多重继承的vptr数量 → 纯虚函数与接口类

### 菱形继承与虚继承 [频率: ⭐⭐⭐⭐]

**核心要点**：菱形继承导致基类数据重复存储和二义性，虚继承通过虚基类表使最派生类只保留一份基类子对象。

**常见面试问法**：
- "菱形继承有什么问题？虚继承怎么解决的？"
- "虚继承的内存布局是怎样的？"

**回答框架**：
1. 菱形继承问题：D继承B和C，B和C都继承A，D中有两份A的成员，访问二义性
2. 虚继承解决：B和C虚继承A，D中只保留一份A子对象，由D负责构造A
3. 内存布局：虚继承的类包含虚基类表指针（vbptr），指向虚基类表，表中记录虚基类子对象的偏移量
4. 访问虚基类成员：通过vbptr查表得到偏移量，间接访问，比普通继承多一次间接寻址
5. 替代方案：优先使用组合替代多重继承，或使用接口类（纯虚函数无数据成员）

**延伸考点**：虚继承 → 虚基类表 → 多重继承的指针调整（this偏移）

### RAII机制 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：RAII（Resource Acquisition Is Initialization）利用对象生命周期管理资源，构造时获取资源，析构时释放资源，确保异常安全。

**常见面试问法**：
- "什么是RAII？为什么C++需要它？"
- "如何用RAII管理互斥锁？"

**回答框架**：
1. 核心思想：资源获取即初始化，资源释放即析构，将资源生命周期绑定到栈对象
2. 为什么需要：C++异常可能导致跳过资源释放代码，RAII通过栈展开自动调用析构函数保证释放
3. 典型应用：智能指针管理内存、lock_guard管理互斥锁、fstream管理文件句柄
4. 实现要点：构造函数获取资源，析构函数释放资源，禁止或限制拷贝（或实现深拷贝/移动语义）
5. 与GC对比：RAII是确定性的（离开作用域立即释放），GC是非确定性的（等待回收时机）

**延伸考点**：RAII → 异常安全（基本/强/不抛保证） → 智能指针 → ScopeGuard

### 内存管理 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：C++程序内存分为栈、堆、全局/静态区、常量区和代码段，理解堆栈差异和new/delete机制是内存管理的基础。

**常见面试问法**：
- "堆和栈有什么区别？哪个快？"
- "new/delete和malloc/free有什么区别？"

**回答框架**：
1. 内存布局：代码段 → 常量区 → 全局/静态区 → 堆（向上增长）→ ... → 栈（向下增长）
2. 栈：编译器自动管理，分配释放只需移动栈指针，速度快，空间有限（通常1-8MB）
3. 堆：手动管理（new/delete），需查找空闲链表，可能产生碎片，空间大但分配慢
4. new/delete vs malloc/free：new调用构造函数/delete调用析构函数；new返回类型安全指针；new失败抛bad_alloc（malloc返回NULL）；new可重载；new[]需匹配delete[]
5. 内存对齐：CPU按字长访问内存，未对齐需要多次访问；结构体大小是最大成员对齐数的整数倍；`#pragma pack` 可修改对齐规则

**延伸考点**：内存对齐 → allocator → 内存池 → tcmalloc/jemalloc

### 内存泄漏 [频率: ⭐⭐⭐⭐]

**核心要点**：内存泄漏是动态分配的内存未被释放且无法再访问，常见原因包括裸指针丢失、异常跳过delete、循环引用等。

**常见面试问法**：
- "如何检测和预防内存泄漏？"
- "什么情况下new了但没delete？"

**回答框架**：
1. 常见原因：裸指针丢失引用、异常路径跳过delete、shared_ptr循环引用、基类析构非虚导致派生类资源未释放
2. 检测工具：Valgrind（Linux）、AddressSanitizer（编译期插桩，快）、Visual Studio CRT调试堆（Windows）
3. 预防手段：优先使用智能指针、RAII管理所有资源、遵循"谁分配谁释放"原则
4. 代码审查要点：检查所有new是否有对应delete、检查异常安全、检查基类虚析构
5. 运行时监控：重载operator new/delete记录分配信息、定期检查分配计数

**延伸考点**：内存泄漏 → AddressSanitizer → 智能指针 → RAII

### 指针与引用的区别 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：引用是对象的别名，必须初始化且不可重绑定；指针是存储地址的变量，可以为空、可以重指向。函数参数优先用引用。

**常见面试问法**：
- "指针和引用有什么区别？什么时候用哪个？"
- "引用底层是怎么实现的？"

**回答框架**：
1. 初始化：引用必须初始化且不可为空，指针可以不初始化（野指针危险）可以为nullptr
2. 重绑定：引用初始化后不可改变引用对象，指针可以重新指向其他对象
3. 底层实现：引用本质上是通过指针实现的，但编译器可能优化掉间接访问
4. 使用场景：函数参数传递优先用const引用（避免拷贝且保证不修改）；需要"可选"参数或"延迟绑定"时用指针
5. 运算：指针支持算术运算，引用不支持；sizeof引用得到所引对象大小，sizeof指针得到地址大小
6. 特殊：引用没有顶层const（引用本身不可重绑定），指针有顶层const（指针本身不可改）和底层const（所指对象不可改）

**延伸考点**：引用 → 右值引用 → 转发引用 → const引用延长临时对象生命周期

### static关键字 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：static控制存储位置和可见性，包括静态局部变量、静态成员、静态函数和静态全局变量，各自改变生命周期或作用域。

**常见面试问法**：
- "static有几种用法？分别有什么效果？"
- "静态成员变量在哪儿初始化？为什么？"

**回答框架**：
1. 静态局部变量：存储在全局/静态区，首次执行到声明处初始化，之后保持值不变，作用域仍为函数内
2. 静态全局变量：存储在全局/静态区，作用域限制在当前翻译单元（文件），外部文件不可extern访问
3. 静态成员变量：属于类而非对象，所有实例共享一份，必须在类外定义初始化（const整型可类内初始化）
4. 静态成员函数：属于类而非对象，无this指针，只能访问静态成员，可通过类名直接调用
5. 静态自由函数：作用域限制在当前翻译单元，替代匿名命名空间
6. C++11补充：constexpr静态成员变量可类内初始化；inline静态成员变量（C++17）可类内定义

**延伸考点**：static → 单例模式 → 线程安全的静态局部变量初始化（C++11保证） → static与线程局部存储(thread_local)

### const关键字 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：const表示不可修改，涵盖变量、指针、引用、函数参数、成员函数等场景，是类型安全的重要手段。

**常见面试问法**：
- "const int *p和int * const p有什么区别？"
- "为什么const成员函数中不能修改成员变量？mutable有什么用？"

**回答框架**：
1. const变量：必须初始化，值不可修改；编译器可能将其放入符号表优化（常量折叠）
2. const指针：`const int *p`（指向const的指针，所指值不可改）vs `int * const p`（指针本身不可改）vs `const int * const p`（都不可改）；读法从右往左读
3. const引用：可绑定到右值延长生命周期，常用于函数参数避免拷贝且保证不修改
4. const成员函数：承诺不修改对象成员变量，本质是将this指针类型变为 `const T* const`；mutable成员变量可在const函数中修改
5. const与#define对比：const有类型检查、有作用域、可调试；#define无类型无作用域仅文本替换
6. const与函数重载：顶层const不影响重载（`void f(int)` 和 `void f(const int)` 不构成重载），底层const影响（`void f(int*)` 和 `void f(const int*)` 构成重载）

**延伸考点**：const → constexpr → const_cast → volatile const

## 二、计算机基础

### 进程与线程 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：进程是资源分配的基本单位，线程是CPU调度的基本单位；线程共享进程的地址空间和资源，拥有独立的栈和寄存器。

**常见面试问法**：
- "进程和线程有什么区别？"
- "线程共享了进程的哪些资源？哪些是独占的？"

**回答框架**：
1. 定义：进程是程序执行的实例，拥有独立地址空间；线程是进程内的执行流
2. 共享资源：代码段、数据段、堆、文件描述符表、信号处理、当前工作目录
3. 独占资源：线程ID、寄存器、栈、errno、信号掩码、线程局部存储
4. 上下文切换：线程切换只需保存/恢复寄存器和栈指针（同地址空间），进程切换还需切换页表和TLB（开销大）
5. 通信方式：线程间通过共享内存直接通信（需同步）；进程间需IPC机制
6. 协程补充：用户态轻量级线程，由程序控制调度，切换无需内核介入，开销极小

**延伸考点**：进程线程 → 协程 → 线程池 → C++20 std::jthread

### 进程间通信（IPC） [频率: ⭐⭐⭐⭐]

**核心要点**：IPC机制包括管道、消息队列、共享内存、信号量、信号、Socket等，各有适用场景；共享内存最快但需同步。

**常见面试问法**：
- "进程间通信有哪些方式？各自的优缺点？"
- "共享内存为什么最快？怎么保证同步？"

**回答框架**：
1. 管道：半双工，父子进程间使用；命名管道可用于无亲缘关系进程；数据流式读取
2. 消息队列：内核中的消息链表，按类型读取，有拷贝开销
3. 共享内存：映射同一块物理内存到不同进程地址空间，零拷贝最快；需配合信号量/互斥锁同步
4. 信号量：计数器，用于进程间同步，P/V操作
5. 信号：异步通知机制，只能传递信号编号，信息量有限
6. Socket：可跨主机通信，TCP可靠/UDP快速；本地Socket（Unix Domain Socket）同主机零拷贝
7. 选型：同主机大数据量用共享内存，跨主机用Socket，简单通知用信号

**延伸考点**：IPC → Binder（Android） → D-Bus → 共享内存的缓存一致性问题

### 死锁 [频率: ⭐⭐⭐⭐]

**核心要点**：死锁是两个或多个进程/线程互相等待对方持有的资源，导致都无法继续执行；需四个条件同时满足才发生。

**常见面试问法**：
- "死锁的四个必要条件是什么？如何破坏？"
- "如何检测和预防死锁？"

**回答框架**：
1. 四个必要条件：互斥（资源独占）、持有并等待（持有资源同时等待其他）、不可抢占（资源不可强制剥夺）、循环等待（等待链成环）
2. 预防策略：破坏互斥（SPooling技术，有限）、破坏持有并等待（一次性申请所有资源）、破坏不可抢占（超时释放）、破坏循环等待（资源有序分配法）
3. 避免：银行家算法，在分配前检查是否会导致不安全状态
4. 检测：资源分配图检测环路；线程dump分析等待关系
5. 恢复：终止进程、抢占资源、回滚到检查点
6. 实践经验：锁的获取顺序全局一致、使用std::lock同时获取多把锁、设置超时（try_lock_for）

**延伸考点**：死锁 → 银行家算法 → 分布式死锁 → 活锁

### 虚拟内存 [频率: ⭐⭐⭐⭐]

**核心要点**：虚拟内存为每个进程提供独立的虚拟地址空间，通过页表映射到物理内存，实现进程隔离、内存保护和按需分配。

**常见面试问法**：
- "虚拟内存是怎么工作的？"
- "缺页中断的处理流程是什么？"

**回答框架**：
1. 核心机制：CPU发出虚拟地址 → MMU查页表 → 转换为物理地址 → 访问内存
2. 页表结构：多级页表减少空间开销，典型为4级（PGD→PUD→PMD→PTE）；TLB缓存近期转换加速
3. 缺页中断：访问的虚拟页不在物理内存中 → 触发缺页中断 → 内核从磁盘换入页面 → 更新页表 → 重新执行指令
4. 页面置换算法：FIFO（简单但Belady异常）、LRU（效果好但开销大）、Clock（近似LRU）、LFU
5. 写时复制（COW）：fork后父子进程共享物理页，标记只读，写入时才拷贝，节省内存
6. 内存映射文件：mmap将文件映射到虚拟地址空间，按需读入，适合大文件随机访问

**延伸考点**：虚拟内存 → TLB → 大页（HugePages） → COW → mmap

### 进程调度算法 [频率: ⭐⭐⭐]

**核心要点**：调度算法决定哪个就绪进程获得CPU，需平衡吞吐量、响应时间、公平性；现代OS采用多级反馈队列。

**常见面试问法**：
- "常见的进程调度算法有哪些？各适用于什么场景？"
- "Linux用的什么调度算法？"

**回答框架**：
1. FCFS：先来先服务，简单但不利于短作业
2. SJF：最短作业优先，平均等待时间最小但可能饥饿长作业
3. SRF：最短剩余时间优先，SJF的抢占版本
4. 时间片轮转：公平，时间片太短切换开销大，太长响应慢
5. 优先级调度：可抢占/不可抢占，低优先级可能饥饿（需老化机制）
6. 多级反馈队列：多个优先级队列，时间片随优先级降低而增大，新进程入高优先级队列，用完时间片降级；综合效果好
7. Linux CFS：完全公平调度器，虚拟运行时间小的优先调度，O(1)时间选择

**延伸考点**：调度算法 → CFS → 实时调度 → CPU亲和性

### 用户态与内核态 [频率: ⭐⭐⭐⭐]

**核心要点**：用户态运行应用程序，权限受限；内核态运行OS内核，可访问所有资源；切换通过系统调用、中断、异常触发。

**常见面试问法**：
- "用户态和内核态有什么区别？怎么切换？"
- "系统调用的开销在哪里？"

**回答框架**：
1. 区别：用户态只能访问用户空间（0-3G），不能执行特权指令；内核态可访问所有内存和硬件
2. 切换触发：系统调用（主动）、中断（外部设备）、异常（CPU检测到错误）
3. 切换过程：保存用户态上下文（寄存器入栈）→ 切换栈到内核栈 → 执行内核代码 → 恢复上下文 → 返回用户态
4. 开销来源：保存/恢复寄存器、切换栈、TLB刷新（可能）、指令流水线冲刷、内核态执行逻辑本身
5. 优化：减少系统调用次数（批量操作）、使用vDSO加速部分系统调用、用户态IO（io_uring）

**延伸考点**：用户态内核态 → 系统调用 → 中断 → io_uring → DPDK用户态协议栈

### TCP三次握手 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：三次握手建立可靠连接，客户端和服务端各确认对方的发送和接收能力正常，同时同步初始序列号。

**常见面试问法**：
- "TCP为什么要三次握手？两次不行吗？"
- "第三次握手丢失会怎样？"

**回答框架**：
1. 流程：客户端发SYN → 服务端回SYN+ACK → 客户端发ACK → 连接建立
2. 为什么三次：两次握手无法确认客户端的接收能力；历史SYN可能导致服务端建立无效连接浪费资源
3. 状态变迁：CLOSED → SYN_SENT → ESTABLISHED（客户端）；LISTEN → SYN_RCVD → ESTABLISHED（服务端）
4. 第三次握手丢失：服务端重传SYN+ACK（默认5次，间隔翻倍），超时后关闭连接
5. 初始序列号：基于时钟和连接四元组生成，随机化防止TCP伪造攻击
6. SYN Flood攻击：发送大量SYN不完成握手，耗尽服务端半连接队列；防御：SYN Cookie、增大队列、减小超时

**延伸考点**：三次握手 → SYN Flood → SYN Cookie → TCP Fast Open

### TCP四次挥手 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：四次挥手因为TCP全双工，每个方向需单独关闭；主动关闭方进入TIME_WAIT等待2MSL确保对方收到最终ACK。

**常见面试问法**：
- "为什么需要TIME_WAIT？持续多久？"
- "为什么是四次挥手而不是三次？"

**回答框架**：
1. 流程：主动方发FIN → 被动方回ACK → 被动方发FIN → 主动方回ACK
2. 为什么四次：被动方收到FIN后可能还有数据未发完，需先ACK再等数据发完才发FIN，所以FIN和ACK分两次
3. TIME_WAIT原因：确保最终ACK到达（若丢失对方重传FIN）；等待网络中残留报文消失（避免新连接收到旧数据）
4. TIME_WAIT时长：2MSL（Maximum Segment Lifetime），通常60秒或120秒
5. TIME_WAIT过多：大量短连接导致端口耗尽；解决：长连接复用、SO_REUSEADDR、内核参数调优（tcp_tw_reuse）
6. CLOSE_WAIT过多：说明被动方未正确关闭连接，通常是代码bug（未调用close）

**延伸考点**：四次挥手 → TIME_WAIT → 保活机制 → RST报文 → 半关闭

### epoll [频率: ⭐⭐⭐⭐⭐]

**核心要点**：epoll是Linux高效IO多路复用机制，基于事件通知，只在活跃fd上回调，时间复杂度O(1)，适合高并发场景。

**常见面试问法**：
- "epoll和select/poll有什么区别？为什么epoll更高效？"
- "epoll的LT和ET模式有什么区别？"

**回答框架**：
1. select缺陷：fd集合大小受限（1024）、每次调用需全量拷贝fd集、O(n)遍历所有fd
2. poll改进：用链表无fd数量限制，但仍O(n)遍历
3. epoll优势：红黑树管理fd（增删O(logn)）、就绪链表只返回活跃fd（O(活跃数)）、mmap共享内存减少拷贝、无fd数量限制
4. LT模式（水平触发）：缓冲区有数据就一直通知，编程简单，支持阻塞/非阻塞IO
5. ET模式（边缘触发）：缓冲区从无数据变为有数据时才通知一次，必须非阻塞IO+循环读至EAGAIN，效率更高但编程复杂
6. 使用场景：少量连接用select/poll即可；大量连接中只有少量活跃用epoll（如Web服务器、聊天服务器）

**延伸考点**：epoll → Reactor模式 → io_uring → C10K问题

### HTTP与HTTPS [频率: ⭐⭐⭐⭐⭐]

**核心要点**：HTTP是明文传输的应用层协议，HTTPS在HTTP和TCP之间加入TLS层，通过加密、证书验证和完整性校验保证安全。

**常见面试问法**：
- "HTTPS的握手过程是怎样的？"
- "HTTP/1.1、HTTP/2、HTTP/3有什么区别？"

**回答框架**：
1. HTTP/1.1：持久连接（Keep-Alive）、管道化（有队头阻塞）、Host头支持虚拟主机
2. HTTPS = HTTP + TLS：客户端发ClientHello（支持的加密套件）→ 服务端回ServerHello + 证书 → 客户端验证证书 → 密钥交换（RSA/ECDHE）→ 协商出对称密钥 → 加密通信
3. TLS 1.2 vs 1.3：1.3减少一个RTT（0-RTT恢复），移除不安全加密套件，强制PFS
4. HTTP/2：二进制帧、多路复用（解决HTTP层队头阻塞）、头部压缩（HPACK）、服务端推送、仍基于TCP
5. HTTP/3：基于QUIC（UDP），解决TCP层队头阻塞，0-RTT建连，连接迁移（用Connection ID而非四元组标识连接）
6. 证书验证链：根CA → 中间CA → 服务器证书，逐级验证签名

**延伸考点**：HTTPS → TLS 1.3 → QUIC → HTTP/3 → 证书体系

### DNS解析 [频率: ⭐⭐⭐⭐]

**核心要点**：DNS将域名解析为IP地址，采用分层递归查询机制，涉及浏览器缓存、OS缓存、本地DNS、根/顶级/权威DNS服务器。

**常见面试问法**：
- "输入URL后DNS解析的完整流程？"
- "DNS用的是TCP还是UDP？"

**回答框架**：
1. 解析流程：浏览器缓存 → OS缓存（hosts文件）→ 本地DNS服务器 → 根DNS → 顶级域DNS → 权威DNS → 返回IP
2. 递归查询 vs 迭代查询：客户端到本地DNS是递归，本地DNS到各级DNS是迭代
3. 传输协议：通常用UDP（53端口），响应超过512字节或区域传送用TCP
4. DNS缓存：每级都有TTL控制缓存时间，减少查询延迟
5. DNS劫持/污染：伪造DNS响应返回错误IP；防御：DNSSEC、DoH（DNS over HTTPS）、DoT（DNS over TLS）
6. DNS负载均衡：一个域名对应多个IP，DNS轮询返回不同IP实现简单负载均衡

**延伸考点**：DNS → DNS劫持 → DNSSEC → CDN调度 → 服务发现

### MySQL索引 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：MySQL InnoDB使用B+树索引，聚簇索引存储完整行数据，二级索引存储主键值，查询时可能回表或使用覆盖索引。

**常见面试问法**：
- "为什么MySQL用B+树而不是B树或红黑树？"
- "什么是回表？怎么避免？"

**回答框架**：
1. B+树优势：非叶子节点只存键值，扇出大树高低（3-4层存千万数据）；叶子节点链表连接支持范围查询；查询稳定O(logn)
2. 对比B树：B树每个节点存数据导致扇出小树更高，范围查询需中序遍历
3. 对比红黑树：树高太大（log₂级别），磁盘IO次数多
4. 对比Hash：Hash不支持范围查询和排序
5. 聚簇索引：叶子节点存完整行数据，一个表只有一个（通常是主键）
6. 二级索引：叶子节点存主键值，查询非索引列需回表（查聚簇索引）
7. 覆盖索引：查询列都在索引中，无需回表；explain中Extra显示Using index
8. 最左前缀原则：联合索引(a,b,c)可匹配a、(a,b)、(a,b,c)，不可跳过左列

**延伸考点**：索引 → 覆盖索引 → 索引下推（ICP） → 索引失效场景 → 慢查询优化

### 事务ACID与MVCC [频率: ⭐⭐⭐⭐⭐]

**核心要点**：ACID是事务的四个特性，MVCC是InnoDB实现读写不冲突的并发控制机制，通过undo log版本链和ReadView实现。

**常见面试问法**：
- "MVCC是怎么实现的？"
- "MySQL的隔离级别有哪些？各自解决什么问题？"

**回答框架**：
1. ACID：原子性（undo log）、一致性（是目标，由其他三个保证）、隔离性（锁+MVCC）、持久性（redo log）
2. 隔离级别：读未提交（脏读）→ 读已提交（不可重复读）→ 可重复读（幻读，InnoDB通过间隙锁解决）→ 串行化（无并发问题但性能差）
3. MVCC实现：每行有隐藏列（trx_id事务ID、roll_pointer回滚指针）→ 修改时旧版本通过roll_pointer形成undo log版本链 → 读取时根据ReadView判断可见性
4. ReadView：记录当前活跃事务ID列表（m_ids）、最小事务ID（min_trx_id）、下一个分配的事务ID（max_trx_id）
5. 可见性判断：trx_id < min_trx_id → 可见；trx_id >= max_trx_id → 不可见；min_trx_id <= trx_id < max_trx_id → 不在m_ids中可见
6. RC vs RR：RC每次SELECT创建新ReadView，RR只在首次SELECT创建，所以RR可重复读

**延伸考点**：MVCC → 间隙锁 → Next-Key Lock → 快照读与当前读

### Redis数据类型与持久化 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：Redis有5种基本数据类型和多种扩展类型，持久化支持RDB快照和AOF日志，各有优缺点，生产环境常混合使用。

**常见面试问法**：
- "Redis有哪些数据类型？底层怎么实现的？"
- "RDB和AOF有什么区别？怎么选？"

**回答框架**：
1. 五种基本类型：String（SDS）、List（quicklist=ziplist+linkedlist）、Hash（ziplist/hashtable）、Set（intset/hashtable）、ZSet（ziplist/skiplist+hashtable）
2. 扩展类型：HyperLogLog（基数统计）、Bitmap（位操作）、GeoSpatial（地理位置）、Stream（消息队列）
3. RDB：fork子进程将内存快照写入二进制文件，恢复快、文件紧凑；但非实时可能丢数据，fork大内存时耗时
4. AOF：追加写命令到日志文件，实时性好（everysec每秒刷盘最多丢1秒）；但文件大、恢复慢
5. AOF重写：fork子进程重写为最小命令集，期间新命令同时写旧AOF和重写缓冲区，重写完成后替换
6. 混合持久化（4.0+）：RDB存全量 + AOF存增量，兼顾恢复速度和数据安全
7. 底层编码优化：小数据量用ziplist（紧凑连续内存），大数据量用标准结构；通过 `OBJECT ENCODING` 查看

**延伸考点**：Redis数据类型 → 跳表 → Redis 7.0 listpack → 大Key问题

### Redis缓存问题 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：缓存穿透（查不存在的数据）、缓存击穿（热点key过期）、缓存雪崩（大量key同时过期）是三大经典问题，各有对应解决方案。

**常见面试问法**：
- "缓存穿透、击穿、雪崩分别是什么？怎么解决？"
- "缓存和数据库一致性怎么保证？"

**回答框架**：
1. 缓存穿透：查询不存在的数据，绕过缓存直达数据库；解决：布隆过滤器拦截、缓存空值（设短TTL）、参数校验
2. 缓存击穿：热点key过期瞬间大量请求打到数据库；解决：互斥锁（setnx）只允许一个请求重建缓存、热点key永不过期（逻辑过期）
3. 缓存雪崩：大量key同时过期或Redis宕机；解决：过期时间加随机偏移、多级缓存、Redis集群高可用、限流降级
4. 缓存与数据库一致性：先更新数据库再删缓存（Cache Aside Pattern）；延迟双删防止旧缓存被读请求回填；通过binlog异步删除保证最终一致
5. 布隆过滤器：位数组+多个Hash函数，判断不存在则一定不存在，判断存在可能误判；不支持删除（Counting Bloom Filter支持）

**延伸考点**：缓存问题 → 布隆过滤器 → 分布式锁 → Canal监听binlog → 本地缓存+Caffeine

## 三、设计模式

### 单例模式 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：保证一个类只有一个实例并提供全局访问点，C++11后最推荐Meyers' Singleton（静态局部变量），线程安全且简洁。

**常见面试问法**：
- "写一个线程安全的单例模式"
- "懒汉和饿汉有什么区别？"

**回答框架**：
1. 饿汉：程序启动时即创建实例（静态成员变量初始化），线程安全但可能浪费资源
2. 懒汉：首次使用时创建实例，需处理线程安全
3. 线程安全懒汉（C++11前）：双重检查锁定（DCLP）+ volatile，但存在指令重排问题
4. Meyers' Singleton（C++11推荐）：静态局部变量在首次执行时初始化，C++11保证线程安全且无竞态
5. 防止特殊操作：删除拷贝构造和赋值运算符、将构造函数私有化
6. 单例的缺陷：全局状态难以测试、隐藏依赖、扩展困难；应谨慎使用

**延伸考点**：单例 → DCLP与memory barrier → 依赖注入 → 控制反转

### 工厂模式 [频率: ⭐⭐⭐⭐]

**核心要点**：工厂模式将对象创建与使用解耦，简单工厂用分支逻辑、工厂方法用多态、抽象工厂用于产品族。

**常见面试问法**：
- "简单工厂、工厂方法、抽象工厂有什么区别？"
- "什么时候该用工厂模式？"

**回答框架**：
1. 简单工厂：一个工厂类根据参数创建不同产品，违反开闭原则（新增产品需修改工厂类）
2. 工厂方法：定义工厂接口，每个产品对应一个工厂子类，符合开闭原则但类数量增多
3. 抽象工厂：创建一族相关产品（如不同风格的UI组件），新增产品族容易但新增产品类型困难
4. 选择依据：产品少且稳定用简单工厂；产品经常扩展用工厂方法；需要创建产品族用抽象工厂
5. C++实现要点：工厂返回unique_ptr管理生命周期、用override明确重写、注册机制替代switch-case

**延伸考点**：工厂模式 → 开闭原则 → 依赖倒置 → 抽象工厂与建造者模式对比

### 观察者模式 [频率: ⭐⭐⭐⭐]

**核心要点**：定义对象间一对多的依赖关系，当一个对象状态改变时自动通知所有依赖对象；是事件驱动架构的基础。

**常见面试问法**：
- "观察者模式怎么实现？有什么注意点？"
- "观察者模式和发布订阅模式有什么区别？"

**回答框架**：
1. 角色：Subject（被观察者）维护观察者列表，提供attach/detach/notify方法；Observer（观察者）定义update接口
2. 推模型 vs 拉模型：推模型Subject主动传递数据，拉模型Observer主动获取
3. 观察者 vs 发布订阅：观察者模式Subject和Observer直接通信；发布订阅模式通过Event Channel解耦，发送者和接收者互不知道对方
4. 注意事项：notify时观察者可能detach导致迭代器失效（需复制列表或标记删除）、循环依赖导致无限通知、观察者执行时间影响Subject
5. C++实现：用std::function/std::bind替代虚函数更灵活；用weak_ptr避免观察者生命周期被Subject延长

**延伸考点**：观察者 → 发布订阅 → 事件总线 → 响应式编程

### 策略模式 [频率: ⭐⭐⭐⭐]

**核心要点**：定义一系列算法，将每个算法封装起来并使它们可以互换，使算法的变化独立于使用算法的客户端。

**常见面试问法**：
- "策略模式怎么消除if-else？"
- "策略模式和状态模式有什么区别？"

**回答框架**：
1. 结构：Context持有Strategy引用，Strategy定义接口，ConcreteStrategy实现具体算法
2. 消除if-else：用策略注册表（map<string, Strategy*>）替代条件分支，运行时按key查找对应策略
3. C++实现：构造时注入策略（构造函数传参）、运行时切换策略（setter方法）、用std::function简化轻量策略
4. 策略 vs 状态：策略模式客户端选择策略，策略之间独立；状态模式状态自动切换，状态之间有关联
5. 典型场景：支付方式选择、排序算法选择、压缩算法选择、日志格式化策略

**延伸考点**：策略模式 → 状态模式 → 模板方法模式 → 函数式编程中的策略

### 适配器模式 [频率: ⭐⭐⭐]

**核心要点**：将一个类的接口转换为客户端期望的另一个接口，使原本不兼容的类可以协同工作；分为类适配器和对象适配器。

**常见面试问法**：
- "适配器模式怎么用？类适配器和对象适配器有什么区别？"
- "适配器模式和装饰器模式有什么区别？"

**回答框架**：
1. 类适配器：通过多重继承同时继承Target接口和Adaptee实现，C++中较少使用
2. 对象适配器：持有Adaptee对象，在Target接口方法中委托调用Adaptee的方法，更灵活推荐使用
3. 适配器 vs 装饰器：适配器改变接口使不兼容的类协作，装饰器增强功能但不改变接口
4. 适配器 vs 外观：适配器是接口转换（1:1），外观是简化接口（多:1）
5. 典型场景：集成第三方库（接口不匹配）、旧系统改造、STL中stack/deque/priority_queue都是容器适配器

**延伸考点**：适配器 → 装饰器 → 外观模式 → STL容器适配器

## 四、项目高频追问

### 为什么选择这个技术方案？对比了哪些？ [频率: ⭐⭐⭐⭐⭐]

**核心要点**：面试官考察技术选型的思考过程，需要展示对备选方案的深入了解和基于场景的权衡能力。

**常见面试问法**：
- "为什么用Redis而不是Memcached？"
- "为什么选gRPC而不是REST？"

**回答框架**：
1. 列出2-3个备选方案及其核心差异
2. 从性能、复杂度、团队熟悉度、社区生态、可维护性等维度对比
3. 结合项目具体场景（数据量、QPS、延迟要求）说明选择理由
4. 坦诚所选方案的不足及应对措施

**延伸考点**：技术选型 → 权衡分析 → 系统设计

### 遇到最大的坑是什么？怎么解决的？ [频率: ⭐⭐⭐⭐⭐]

**核心要点**：考察问题定位能力和解决思路，需要展示从发现到定位到解决的完整过程。

**常见面试问法**：
- "线上出过什么问题？怎么排查的？"
- "遇到过最难调的bug是什么？"

**回答框架**：
1. 描述问题现象（监控告警/用户反馈）
2. 排查过程：日志分析 → 链路追踪 → 抓包/性能剖析 → 定位根因
3. 解决方案：临时止血 → 根本修复 → 验证
4. 复盘总结：增加什么监控、改进什么流程、沉淀什么工具

**延伸考点**：问题排查 → 线上运维 → 可观测性

### QPS/性能数据是多少？怎么测的？ [频率: ⭐⭐⭐⭐]

**核心要点**：考察对系统性能的量化认知和压测方法，不能只说"很快"必须有数据。

**常见面试问法**：
- "你的服务QPS是多少？怎么测的？"
- "P99延迟是多少？怎么优化的？"

**回答框架**：
1. 给出具体数据：QPS、P50/P99延迟、错误率
2. 压测方法：工具（wrk/ab/JMeter）、压测场景、逐步加压策略
3. 瓶颈定位：CPU/IO/锁/网络，火焰图/pprof分析
4. 优化措施及效果：优化前后对比数据

**延伸考点**：性能测试 → 火焰图 → 全链路压测 → 容量规划

### 如果重新做会怎么优化？ [频率: ⭐⭐⭐⭐]

**核心要点**：考察技术视野和反思能力，展示对系统演进方向的思考。

**常见面试问法**：
- "现在回头看有什么可以改进的？"
- "如果流量增长10倍怎么办？"

**回答框架**：
1. 架构层面：微服务拆分、引入消息队列解耦、读写分离
2. 性能层面：缓存策略优化、连接池调优、异步化改造
3. 可靠性层面：熔断降级、限流策略、灰度发布
4. 工程层面：自动化测试覆盖率、CI/CD流水线、监控告警完善

**延伸考点**：系统演进 → 高可用设计 → 技术债务管理

### 项目中的难点和亮点 [频率: ⭐⭐⭐⭐⭐]

**核心要点**：难点展示技术深度，亮点展示技术广度和创新性，需要具体且有数据支撑。

**常见面试问法**：
- "项目中最有技术含量的部分是什么？"
- "你在这个项目中的主要贡献是什么？"

**回答框架**：
1. 难点：描述技术挑战（而非业务复杂度）→ 分析为什么难 → 解决思路和方案 → 最终效果
2. 亮点：创新点（技术方案/工具/流程改进）→ 量化收益（性能提升X%、成本降低Y%）→ 可复用性
3. 避免泛泛而谈，用STAR法则（Situation-Task-Action-Result）组织回答
4. 诚实说明团队协作中的角色和自己的具体贡献

**延伸考点**：STAR法则 → 技术影响力 → 工程素养

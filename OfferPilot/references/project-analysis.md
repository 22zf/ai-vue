# 用户项目分析指南

根据用户提供的项目路径或项目描述，分析项目并生成针对性面试题和包装建议。

---

## 分析流程

### Step 1: 项目信息收集

| 输入方式 | 处理方法 |
|----------|----------|
| 用户提供项目路径 | 读取项目代码，自动分析 |
| 用户描述项目 | 从描述中提取关键信息 |
| 用户提供 GitHub 链接 | 提示用户描述项目核心内容 |

### Step 2: 自动分析（有项目路径时）

#### 2.1 识别技术栈

| 标识文件 | 识别技术 | 对应岗位 |
|----------|----------|----------|
| `package.json` | Node.js / React / Vue / Angular | 前端 |
| `pom.xml` / `build.gradle` | Java / Spring Boot | Java后端 |
| `CMakeLists.txt` / `Makefile` / `.cpp` / `.h` | C/C++ | C++后端 |
| `requirements.txt` / `setup.py` / `pyproject.toml` | Python / ML/DL | AI算法 / 数据分析 |
| `go.mod` | Go | 后端通用 |
| `Dockerfile` / `docker-compose.yml` / `*.yaml` (k8s) | Docker / K8s | 云计算运维 |
| `*.kt` / `*.java` (Android) | Android | 移动端 |
| `*.swift` / `*.xcodeproj` | iOS | 移动端 |
| `pubspec.yaml` | Flutter / Dart | 移动端 |
| `pytest.ini` / `tox.ini` / `conftest.py` | 测试框架 | 测试开发 |
| `Cargo.toml` | Rust | C++后端(通用) |

#### 2.2 识别项目结构

```
分析目录布局，识别：
- 入口文件（main.go / App.jsx / Application.java 等）
- 路由/控制器层 → 了解功能模块
- 服务/业务逻辑层 → 了解核心逻辑
- 数据层/模型层 → 了解数据设计
- 配置文件 → 了解基础设施（数据库/缓存/MQ）
- 测试目录 → 了解测试覆盖
- CI/CD配置 → 了解部署流程
```

#### 2.3 识别技术亮点

```
从代码中寻找：
- 设计模式的使用（工厂/策略/观察者等）
- 性能优化手段（缓存/异步/批处理/连接池等）
- 架构选择（微服务/单体/分层等）
- 中间件使用（Redis/MQ/ES等）
- 安全措施（认证/鉴权/加密等）
- 监控和日志方案
```

### Step 3: 生成面试题

基于项目分析结果，从以下5个维度生成面试题：

#### 维度1: 技术栈八股题

基于项目用到的技术，从对应知识库中筛选高频考点。

```
示例：项目用了 Spring Boot + Redis + MySQL
→ 出题：Spring IoC原理、Redis缓存穿透、MySQL索引优化
```

#### 维度2: 项目实现深挖题

基于项目代码的具体实现，追问细节。

```
示例问题：
- "你项目里的XX模块是怎么实现的？"
- "为什么选择XX方案而不是YY？"
- "这个接口的并发量是多少？怎么处理的？"
- "这里的异常处理是怎么做的？"
```

#### 维度3: 架构设计题

基于项目整体架构，考察设计思维。

```
示例问题：
- "如果用户量增长10倍，你的架构需要怎么改？"
- "为什么选择微服务/单体？有什么取舍？"
- "数据库是怎么分库分表的？"
- "服务间通信用的什么方式？为什么？"
```

#### 维度4: 难点与优化题

基于项目中的复杂实现，考察问题解决能力。

```
示例问题：
- "项目中最难的技术点是什么？怎么解决的？"
- "遇到过什么线上问题？怎么排查的？"
- "有没有做过性能优化？效果如何？"
- "如果让你重新设计这个模块，会怎么改？"
```

#### 维度5: 场景延伸题

基于项目场景，考察应变能力。

```
示例问题：
- "如果Redis挂了，你的系统会怎样？"
- "如果MQ消息积压了100万条，怎么处理？"
- "如何保证接口的幂等性？"
- "如何做灰度发布？"
```

```
1. **亮点1**：XX（技术点 + 解决的问题 + 效果）
2. **亮点2**：XX
3. **亮点3**：XX

### 💬 面试叙述建议（30秒版）
> "这个项目是XX，主要解决XX问题。我负责XX模块，核心用到了XX技术，

1. "为什么选XX技术？" → 建议回答方向：对比2-3个方案的优劣

[基于5个维度各出2题]

- 需要补充量化数据的点：...
- 可能被质疑的地方：...

```

## 不同项目类型的分析重点

### Web 后端项目

```
重点分析：
- API设计（RESTful？版本控制？）
- 数据库设计（表结构？索引？分库分表？）
- 缓存策略（缓存什么？更新策略？）
- 并发处理（锁？队列？异步？）
- 鉴权方案（JWT？Session？OAuth？）
- 日志和监控
```

### 前端项目

```
重点分析：
- 组件设计（复用性？抽象层级？）
- 状态管理方案（为什么选这个？）
- 性能优化（懒加载？虚拟列表？缓存？）
- 构建配置（Webpack/Vite配置亮点？）
- 类型安全（TypeScript使用深度？）
- 测试覆盖
```

### AI/算法项目

```
重点分析：
- 数据处理流程（清洗？增强？标注？）
- 模型选择（为什么选这个模型？对比了哪些？）
- 训练策略（超参？调优方法？）
- 评估指标（为什么选这个指标？）
- 推理优化（量化？蒸馏？加速？）
- 线上效果（A/B测试？业务指标提升？）
```

### 系统底层项目

```
重点分析：
- 架构设计（为什么这么设计？）
- 核心算法实现（时间/空间复杂度？）
- 并发模型（线程模型？锁策略？）
- 内存管理（分配策略？泄漏检测？）
- 性能基准（QPS？延迟？对比？）
```

---

## 项目分析注意事项

1. **不要只看技术栈**，要深入代码逻辑找亮点
2. **关注非常规选择**，面试官最爱问"为什么不用XX"
3. **量化一切**，没有数据的要建议用户补充
4. **找项目的"故事性"**，面试是讲故事不是报菜名
5. **诚实评估**，不要帮用户过度包装，面试官一问就露馅

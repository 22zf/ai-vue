# MLOps / 机器学习工程 面试八股文

## 一、ML 系统基础

### ML 系统与传统软件区别 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：ML系统以数据为核心驱动，具有实验性强、不确定性高、持续退化等本质特征
**常见面试问法**：
- "ML系统和传统软件系统有什么本质区别？"
- "为什么ML系统比传统软件更难维护？"
**回答框架**：
1. 先说结论：ML系统的行为由数据和模型共同决定，而非显式编码的逻辑
2. 展开原理：数据依赖导致系统行为不确定、模型性能随时间退化、实验性开发流程
3. 对比相关概念：传统软件确定性输入→确定性输出，ML系统概率性输出且对数据分布敏感
4. 举例应用场景：推荐系统随用户偏好变化而退化、欺诈检测需持续对抗新型欺诈手段
**延伸考点**：ML技术债 → ML生命周期

### ML 生命周期 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：ML生命周期是数据→训练→评估→部署→监控→迭代的闭环过程
**常见面试问法**：
- "请描述一个完整的ML项目生命周期"
- "ML生命周期中哪个阶段最容易出问题？"
**回答框架**：
1. 先说结论：ML生命周期是持续迭代的闭环，不是一次性交付
2. 展开原理：问题定义→数据采集与标注→特征工程→模型训练→离线评估→在线评估→部署→监控→触发重训
3. 对比相关概念：传统软件SDLC是线性的，ML生命周期是螺旋上升的
4. 举例应用场景：推荐系统上线后监控CTR下降→触发数据排查→发现特征漂移→重训模型→灰度发布
**延伸考点**：模型监控 → 模型重训练

### ML 技术债 [频率: ⭐⭐⭐⭐]
**核心要点**：ML系统存在大量隐藏的技术债，数据债和模型债往往远超代码债
**常见面试问法**：
- "什么是ML技术债？有哪些类型？"
- "如何管理ML系统中的技术债？"
**回答框架**：
1. 先说结论：ML技术债是系统维护中积累的隐性成本，数据相关债务占主导
2. 展开原理：数据债（数据质量差、标注不一致）、模型债（过拟合、概念漂移）、工程债（管线耦合、配置散落）、隐藏债（特征纠缠、实验毒化）
3. 对比相关概念：传统技术债主要是代码层面，ML技术债更多在数据和模型层面
4. 举例应用场景：特征管线与训练管线逻辑不一致导致训练-服务偏差
**延伸考点**：数据质量 → 特征监控

## 二、特征工程与特征平台

### 特征存储 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：特征存储统一管理离线训练和在线推理的特征，解决训练-服务偏差问题
**常见面试问法**：
- "请说说Feature Store的原理和作用"
- "Feast和Tecton有什么区别？"
**回答框架**：
1. 先说结论：Feature Store是特征的单一事实来源，提供离线和在线特征的一致访问
2. 展开原理：离线存储（历史特征，用于训练，通常用Parquet/Hive）、在线存储（低延迟特征服务，通常用Redis/DynamoDB）、特征注册（元数据管理、特征发现）
3. 对比相关概念：Feast是开源轻量方案，Tecton是商业全托管方案，Hopsworks提供完整特征平台
4. 举例应用场景：推荐系统中用户画像特征离线用Hive表训练、在线用Redis服务推理请求
**延伸考点**：特征计算 → 训练-服务偏差

### 特征计算 [频率: ⭐⭐⭐⭐]
**核心要点**：特征计算分为批处理、流处理和点查询三种模式，需保证离线在线一致性
**常见面试问法**：
- "如何保证离线特征和在线特征的一致性？"
- "什么是特征回填（backfill）？"
**回答框架**：
1. 先说结论：特征计算的核心挑战是离线训练和在线推理使用同一套计算逻辑
2. 展开原理：批处理（定时计算历史特征，适合慢变特征）、流处理（实时计算增量特征，适合快变特征）、点查询（在线请求时即时计算）、特征回填（用新逻辑重新计算历史特征以避免数据泄露）
3. 对比相关概念：批处理延迟高但数据完整，流处理延迟低但可能丢数据，点查询实时但计算受限
4. 举例应用场景：用户过去7天点击量用批处理、最近5分钟点击量用流处理、当前会话特征用点查询
**延伸考点**：特征存储 → 数据管线编排

### 特征监控 [频率: ⭐⭐⭐⭐]
**核心要点**：特征监控检测数据漂移、特征重要性变化和缺失率异常，保障模型输入质量
**常见面试问法**：
- "如何监控线上特征的质量？"
- "特征漂移和概念漂移有什么区别？"
**回答框架**：
1. 先说结论：特征监控是模型监控的前置防线，在模型性能下降前发现输入问题
2. 展开原理：数据漂移（特征分布变化，用PSI/KS检测）、特征重要性变化（模型依赖的特征发生偏移）、缺失率监控（上游数据源异常导致特征缺失）、特征相关性变化（特征间关系变化影响模型）
3. 对比相关概念：数据漂移是输入分布变化，概念漂移是输入输出关系变化
4. 举例应用场景：监控到用户年龄特征分布PSI>0.2，排查发现数据采集逻辑变更
**延伸考点**：模型监控 → 数据质量

## 三、实验管理

### 实验追踪 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：实验追踪系统化管理每次实验的参数、指标和产物，确保实验可追溯可比较
**常见面试问法**：
- "你们团队怎么管理实验的？用的什么工具？"
- "MLflow和W&B有什么区别？"
**回答框架**：
1. 先说结论：实验追踪记录超参数、度量指标、代码版本、数据版本、模型产物，形成完整实验 lineage
2. 展开原理：参数记录（超参数、配置）、指标记录（训练曲线、评估指标）、产物管理（模型文件、可视化图表）、实验对比（不同实验的指标对比）、协作共享（团队查看和复现实验）
3. 对比相关概念：MLflow开源自托管、W&B云端SaaS且可视化强、Neptune侧重实验对比、ClearML集成CI/CD
4. 举例应用场景：用MLflow Tracking记录100次超参搜索实验，对比AUC找到最优配置
**延伸考点**：实验可复现 → 模型治理

### 超参调优 [频率: ⭐⭐⭐⭐]
**核心要点**：超参调优通过系统化搜索策略找到最优模型配置，平衡搜索效率与调优效果
**常见面试问法**：
- "超参调优有哪些方法？各有什么优缺点？"
- "Optuna的原理是什么？"
**回答框架**：
1. 先说结论：超参调优从暴力搜索到智能搜索，核心是用最少资源找到最优配置
2. 展开原理：Grid Search（穷举、维度灾难）、Random Search（随机采样、效率更高）、Bayesian Optimization（高斯过程建模、利用历史信息）、TPE（树状Parzen估计器、Optuna默认）、ASHA/Successive Halving（早停淘汰、资源高效）、Population Based Training（训练中调参）
3. 对比相关概念：Grid适合低维小空间、Random适合高维大空间、Bayesian适合昂贵评估、ASHA适合大规模并行
4. 举例应用场景：用Optuna的TPE+Pruner对学习率、batch size、dropout进行调优，100次试验找到最优组合
**延伸考点**：实验追踪 → GPU调度

### 实验可复现 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：实验可复现需要锁定随机种子、环境、数据和模型版本四个维度
**常见面试问法**：
- "如何保证实验可复现？"
- "为什么同样的代码跑出不同结果？"
**回答框架**：
1. 先说结论：实验可复现需要同时控制随机性、环境、数据和代码四个变量
2. 展开原理：随机种子（Python/NumPy/PyTorch/CUDA的随机源全部固定）、环境锁定（Docker镜像+pip freeze/conda lock）、数据版本（DVC/Git LFS追踪数据快照）、模型版本（MLflow Registry管理模型版本）、确定性算法（CUDA确定性模式、避免异步操作）
3. 对比相关概念：可复现≠可解释，可复现关注结果一致，可解释关注原因理解
4. 举例应用场景：用DVC锁定训练数据版本+Docker锁定环境+固定所有随机种子，确保3个月后能复现同一模型
**延伸考点**：数据版本 → 模型治理

## 四、模型训练工程

### 分布式训练 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：分布式训练通过数据并行或模型并行将训练任务拆分到多GPU/多节点
**常见面试问法**：
- "数据并行和模型并行有什么区别？"
- "DeepSpeed的ZeRO原理是什么？"
**回答框架**：
1. 先说结论：分布式训练的核心挑战是通信开销和内存效率，不同策略针对不同瓶颈
2. 展开原理：数据并行（每个GPU持完整模型副本、切分数据、AllReduce同步梯度）、模型并行（切分模型到多个GPU、层间或层内并行）、ZeRO（优化器状态/梯度/参数分片，三级优化）、DeepSpeed（ZeRO+卸载+流水线并行）、Megatron-LM（张量并行+流水线并行用于大模型）、Ring AllReduce（环形通信减少带宽瓶颈）
3. 对比相关概念：数据并行通信量大但实现简单，模型并行通信量小但需要拆分模型，混合并行结合两者优势
4. 举例应用场景：训练7B模型用8卡数据并行+ZeRO Stage3，70B模型用Megatron的张量并行+流水线并行
**延伸考点**：训练加速 → GPU调度

### 训练加速 [频率: ⭐⭐⭐⭐]
**核心要点**：训练加速从计算、内存、I/O三个维度优化训练效率
**常见面试问法**：
- "如何加速模型训练？有哪些常用技术？"
- "混合精度训练的原理是什么？会不会影响精度？"
**回答框架**：
1. 先说结论：训练加速是计算效率、内存效率、数据效率的综合优化
2. 展开原理：混合精度（FP16计算+FP32主权重、损失缩放防止梯度下溢）、梯度累积（小batch多次前向后向再更新，等效大batch）、检查点（不保存中间激活、反向时重计算，用时间换空间）、断点续训（保存optimizer state+模型权重+epoch，故障后恢复）、数据预取（GPU计算时CPU准备下一批数据）、编译优化（torch.compile/XLA融合算子）
3. 对比相关概念：混合精度几乎不损失精度但需loss scaling，梯度累积不省计算时间但省显存，检查点省显存但增加计算
4. 举例应用场景：用混合精度+梯度累积+激活检查点，在单卡32G上训练原本需要80G显存的模型
**延伸考点**：分布式训练 → 推理延迟优化

### GPU 调度 [频率: ⭐⭐⭐⭐]
**核心要点**：GPU调度解决多团队共享GPU资源时的分配效率、利用率和成本问题
**常见面试问法**：
- "如何管理团队的GPU资源？"
- "弹性训练和Spot实例怎么用？"
**回答框架**：
1. 先说结论：GPU调度的核心目标是最大化利用率、最小化等待时间和成本
2. 展开原理：K8s GPU调度（设备插件、资源请求/限制、拓扑感知）、弹性训练（节点动态加入/移除、调整batch size和learning rate）、Spot实例（抢占式GPU、低成本但可能中断、需配合检查点）、GPU共享（MPS/MIG、时间片、显存隔离）、优先级调度（训练/推理/开发不同优先级）
3. 对比相关概念：独占GPU利用率低但隔离好，共享GPU利用率高但可能干扰，Spot成本低但需要容错
4. 举例应用场景：训练任务用Spot实例+每100步保存检查点，推理服务用按需实例保证可用性
**延伸考点**：训练加速 → 训练成本优化

## 五、模型部署

### 模型服务 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：模型服务将训练好的模型转化为可调用的推理API，核心权衡延迟、吞吐和成本
**常见面试问法**：
- "模型服务有哪些方案？怎么选型？"
- "Triton和ONNX Runtime有什么区别？"
**回答框架**：
1. 先说结论：模型服务框架的选择取决于模型格式、性能需求和部署环境
2. 展开原理：Triton Inference Server（支持多框架、动态批处理、模型管线）、TensorRT（NVIDIA GPU极致优化、算子融合、INT8量化）、ONNX Runtime（跨平台、CPU/GPU通用、图优化）、模型格式转换（PyTorch→ONNX→TensorRT、转换中的算子兼容性）、动态批处理（将多个请求合并为一个batch提升吞吐）
3. 对比相关概念：Triton是服务框架管理模型生命周期，TensorRT是推理引擎优化计算，ONNX Runtime是轻量跨平台方案
4. 举例应用场景：CV模型转ONNX再用TensorRT优化，部署在Triton上，P99延迟从50ms降到8ms
**延伸考点**：模型压缩 → 推理延迟优化

### 部署策略 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：部署策略控制新模型上线的流量比例和风险范围，平衡验证速度与线上安全
**常见面试问法**：
- "模型上线一般用什么部署策略？"
- "A/B测试和金丝雀发布有什么区别？"
**回答框架**：
1. 先说结论：部署策略的核心是在验证新模型效果的同时最小化线上风险
2. 展开原理：A/B测试（随机分流、统计显著性验证、需足够流量和时间）、金丝雀发布（小比例流量先上线、逐步扩大、快速回滚）、影子模式（新模型并行推理但不返回结果、对比线上模型输出）、多臂老虎机（动态分配流量、探索与利用平衡、自动收敛到最优模型）、蓝绿部署（两套环境切换、零停机、资源翻倍）
3. 对比相关概念：A/B测试重在统计验证，金丝雀重在风险控制，影子模式重在安全对比，MAB重在自动优化
4. 举例应用场景：新推荐模型先影子模式跑一周对比CTR，再金丝雀5%流量观察3天，最后全量发布
**延伸考点**：模型监控 → 模型回滚

### 模型压缩 [频率: ⭐⭐⭐⭐]
**核心要点**：模型压缩通过量化、剪枝、蒸馏降低模型大小和推理成本，尽量保持精度
**常见面试问法**：
- "模型压缩有哪些方法？分别什么原理？"
- "量化和蒸馏有什么区别？"
**回答框架**：
1. 先说结论：模型压缩的目标是用更少资源达到接近原始模型的精度
2. 展开原理：量化（PTQ训练后量化/QAT量化感知训练、FP32→INT8/INT4、权重+激活量化）、剪枝（非结构化剪枝删单个权重/结构化剪枝删整个通道、稀疏推理加速）、知识蒸馏（教师模型指导学生模型、软标签+中间层特征、TinyBERT/DistilBERT）、低秩分解（SVD分解权重矩阵、减少参数量）
3. 对比相关概念：量化不改模型结构只降精度、剪枝删除部分参数需稀疏推理支持、蒸馏训练新小模型
4. 举例应用场景：LLM用GPTQ 4bit量化将70B模型从140GB压到35GB，精度损失<1%
**延伸考点**：模型服务 → 边缘部署

### 边缘部署 [频率: ⭐⭐⭐]
**核心要点**：边缘部署将模型运行在终端设备上，核心挑战是算力、内存和功耗受限
**常见面试问法**：
- "模型怎么部署到手机或嵌入式设备上？"
- "边缘部署有哪些挑战？"
**回答框架**：
1. 先说结论：边缘部署需要在极低资源条件下实现可接受的推理性能
2. 展开原理：TensorFlow Lite（移动端优化、委托GPU/DSP/NNAPI）、Core ML（Apple生态、Neural Engine加速）、ONNX Mobile（跨平台、轻量运行时）、模型适配（量化+剪枝使模型满足设备约束）、硬件感知优化（针对NPU/DSP的算子适配）
3. 对比相关概念：云端部署算力充足但依赖网络，边缘部署离线可用但资源受限
4. 举例应用场景：人脸检测模型用TFLite+INT8量化部署到Android手机，推理耗时从200ms降到30ms
**延伸考点**：模型压缩 → 推理延迟优化

## 六、模型监控与治理

### 模型监控 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：模型监控检测概念漂移和数据漂移，在模型性能退化影响业务前发出告警
**常见面试问法**：
- "线上模型怎么监控？监控哪些指标？"
- "概念漂移和数据漂移有什么区别？怎么检测？"
**回答框架**：
1. 先说结论：模型监控从输入、输出、性能三个层面检测异常，核心是及时发现退化
2. 展开原理：概念漂移（输入输出关系变化，用准确率/线上标签反馈检测）、数据漂移（输入分布变化，用PSI/KS/MMD检测）、预测分布监控（预测类别分布变化、置信度分布偏移）、性能衰减（延迟标签下用代理指标估算）、报警策略（阈值+趋势+组合告警）
3. 对比相关概念：数据漂移是因（输入变了），概念漂移是果（关系变了），性能下降是最终表现
4. 举例应用场景：金融风控模型监控到申请用户特征分布PSI>0.25，触发人工审查发现新型欺诈模式
**延伸考点**：特征监控 → 模型重训练

### 模型重训练 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：模型重训练根据触发策略自动或半自动更新模型，并支持快速回滚
**常见面试问法**：
- "线上模型什么时候需要重训练？怎么重训练？"
- "自动重训练的pipeline怎么设计？"
**回答框架**：
1. 先说结论：重训练需要明确的触发条件、自动化管线和回滚机制
2. 展开原理：触发策略（定时触发/指标触发/数据量触发/人工触发）、自动重训流程（数据拉取→特征计算→模型训练→离线评估→自动/人工审批→灰度发布）、模型回滚（发现新模型不如旧模型时快速切回、需保留历史模型版本）、数据窗口（滚动窗口/扩展窗口选择）、重训频率与成本平衡
3. 对比相关概念：全量重训（用所有历史数据）vs增量训练（只用新数据），全量更稳定但成本高
4. 举例应用场景：推荐模型每天自动重训，AUC低于阈值0.75时告警，低于0.72时自动回滚到上一版本
**延伸考点**：模型监控 → 模型治理

### 模型治理 [频率: ⭐⭐⭐⭐]
**核心要点**：模型治理确保模型从开发到上线的全流程可追溯、可审批、可审计
**常见面试问法**：
- "什么是模型治理？为什么需要？"
- "模型注册和审批流程怎么设计？"
**回答框架**：
1. 先说结论：模型治理是ML系统的合规和风险管理框架，确保模型可追溯可审计
2. 展开原理：模型注册（MLflow Registry/自定义注册中心、模型版本/阶段管理）、审批流程（Staging→Production的审批门禁、评估报告自动生成）、血缘追踪（模型→数据→特征的完整链路、影响分析）、合规审计（模型公平性/可解释性/数据隐私、审计日志）
3. 对比相关概念：模型治理偏管理和合规，MLOps偏工程和自动化，两者互补
4. 举例应用场景：金融模型上线前需通过公平性测试（不同人群差异<5%）+审批流程+血缘文档
**延伸考点**：实验追踪 → 数据安全与合规

## 七、数据管线

### 数据版本 [频率: ⭐⭐⭐⭐]
**核心要点**：数据版本管理追踪数据集的变更历史，支持实验复现和数据回溯
**常见面试问法**：
- "数据怎么版本管理？用的什么工具？"
- "DVC的原理是什么？和Git有什么区别？"
**回答框架**：
1. 先说结论：数据版本管理是ML可复现的基础，需追踪数据与代码的对应关系
2. 展开原理：DVC（Git-like数据版本、元数据存Git、数据存远程存储、dvc push/pull）、LakeFS（数据湖上的Git-like分支/合并/回滚、S3兼容）、数据血缘（数据集→特征→模型的依赖关系、影响分析）、数据快照（训练时锁定数据版本、不可变数据集）
3. 对比相关概念：Git管理小文件文本，DVC管理大文件二进制数据，LakeFS管理数据湖中的表
4. 举例应用场景：用DVC锁定训练数据版本，模型v3对应data_v2，回溯时能精确恢复训练数据
**延伸考点**：实验可复现 → 数据质量

### 数据质量 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：数据质量是ML系统可靠性的基石，需在数据进入管线时进行验证和监控
**常见面试问法**：
- "如何保证训练数据的质量？"
- "Great Expectations是什么？怎么用？"
**回答框架**：
1. 先说结论：数据质量问题的修复成本随流程推进指数增长，需尽早发现尽早拦截
2. 展开原理：Great Expectations（定义数据期望/验证/生成数据文档）、数据验证（Schema验证/范围验证/统计验证/自定义规则）、异常检测（统计方法检测异常值/孤立森林/Z-score）、数据质量监控（缺失率/重复率/分布偏移/时效性）、数据质量门禁（验证不通过阻断下游流程）
3. 对比相关概念：数据验证是主动检查，数据监控是持续观测，数据质量门禁是强制拦截
4. 举例应用场景：用Great Expectations定义"用户年龄在0-120之间"的期望，训练前自动验证，异常数据阻断训练
**延伸考点**：特征监控 → 数据管线编排

### 数据管线编排 [频率: ⭐⭐⭐⭐]
**核心要点**：数据管线编排工具管理数据处理任务的依赖关系、调度和容错
**常见面试问法**：
- "Airflow和Prefect有什么区别？"
- "ML数据管线怎么编排？"
**回答框架**：
1. 先说结论：管线编排解决的是任务依赖管理、调度执行和故障恢复问题
2. 展开原理：Airflow（DAG定义依赖、调度器+执行器、成熟生态、但静态DAG不易动态生成）、Prefect（Python原生、动态DAG、失败重试更灵活）、Dagster（资产驱动、数据血缘内置、类型系统）、Kubeflow Pipelines（K8s原生、ML场景专用、组件化）、容错策略（重试/跳过/降级/告警）
3. 对比相关概念：Airflow偏通用调度、Prefect偏Python友好、Dagster偏数据资产、Kubeflow偏ML场景
4. 举例应用场景：用Airflow编排每日推荐管线：数据采集→特征计算→模型训练→评估→部署，任一步失败告警并重试
**延伸考点**：特征计算 → 模型重训练

## 八、MLOps 平台

### Kubeflow [频率: ⭐⭐⭐⭐]
**核心要点**：Kubeflow是K8s原生的ML平台，覆盖训练、调参、部署全流程
**常见面试问法**：
- "Kubeflow的架构是怎样的？有哪些组件？"
- "Kubeflow和MLflow有什么区别？"
**回答框架**：
1. 先说结论：Kubeflow是面向K8s的ML全流程平台，适合K8s成熟的团队
2. 展开原理：Pipeline（Argo为基础的ML工作流、组件化、可复用）、Notebook（Jupyter开发环境、K8s Pod运行）、Katib（超参调优、支持多种搜索算法）、Serving（KServe模型服务、自动扩缩容、金丝雀部署）、Training Operator（PyTorch/TensorFlow分布式训练Operator）
3. 对比相关概念：Kubeflow是全流程平台偏基础设施，MLflow偏实验管理轻量级，两者常搭配使用
4. 举例应用场景：用Kubeflow Pipeline编排从数据预处理到模型部署的全流程，Katib自动调参，KServe服务模型
**延伸考点**：MLflow → 端到端平台

### MLflow [频率: ⭐⭐⭐⭐⭐]
**核心要点**：MLflow是开源ML生命周期管理平台，核心四大组件覆盖实验到服务
**常见面试问法**：
- "MLflow有哪些组件？分别做什么？"
- "MLflow怎么管理模型版本？"
**回答框架**：
1. 先说结论：MLflow是轻量级ML平台，以实验追踪为核心，扩展到模型管理和服务
2. 展开原理：Tracking（实验追踪、参数/指标/产物记录、支持多种后端存储）、Registry（模型注册中心、版本管理、阶段转换Staging/Production/Archived、审批流程）、Projects（可复现的打包格式、环境管理、支持Docker/Conda）、Serve（模型服务、支持多种框架、实时/批量推理）
3. 对比相关概念：MLflow Tracking是核心最常用，Registry是模型治理基础，Projects和Serve相对轻量
4. 举例应用场景：训练脚本加mlflow.log_param/log_metric自动记录实验，mlflow.register_model注册模型，审批后mlflow.deploy部署服务
**延伸考点**：实验追踪 → 模型治理

### 端到端平台 [频率: ⭐⭐⭐⭐]
**核心要点**：端到端ML平台整合数据、训练、部署全链路，降低ML落地门槛
**常见面试问法**：
- "你们用的什么ML平台？怎么选型的？"
- "自建ML平台和用云平台怎么选？"
**回答框架**：
1. 先说结论：端到端平台的选择取决于团队规模、数据敏感度、定制化需求和成本
2. 展开原理：Vertex AI（GCP原生、AutoML+自定义训练、集成数据平台）、SageMaker（AWS原生、Ground Truth标注+训练+部署、内置算法）、PAI（阿里云、中文生态、PAI-Studio/PAI-DLC/PAI-EAS）、自建平台（基于Kubeflow/MLflow组合、灵活但运维成本高）
3. 对比相关概念：云平台开箱即用但锁定供应商，自建灵活但需要平台工程团队
4. 举例应用场景：中型团队用SageMaker做标注+训练+部署，大型团队自建Kubeflow+MLflow+Feast平台
**延伸考点**：Kubeflow → MLflow

## 九、高频项目方向

### 推荐系统 MLOps
**核心要点**：推荐系统MLOps的核心是特征平台实时性、模型服务高吞吐和AB测试闭环
**常见面试问法**：
- "推荐系统的MLOps怎么做的？"
- "推荐特征平台怎么设计？"
**回答框架**：
1. 先说结论：推荐系统MLOps的难点在于特征实时性要求高、模型更新频繁、效果验证依赖AB测试
2. 展开原理：特征平台（离线特征Hive+在线特征Redis、特征回填保证一致性）、模型服务（多路召回+精排模型、高QPS低延迟、动态批处理）、AB测试（分层实验框架、指标显著性检验、长期效果评估）、模型更新（日级自动重训、增量更新、模型热加载）
3. 对比相关概念：推荐MLOps偏实时特征和AB测试，CV MLOps偏模型优化和推理加速
4. 举例应用场景：推荐系统每天自动重训精排模型，AB测试验证CTR提升>0.5%后全量发布
**延伸考点**：特征存储 → 部署策略 → 模型监控

### CV 模型部署
**核心要点**：CV模型部署的核心挑战是模型大、推理慢、边缘设备资源受限
**常见面试问法**：
- "CV模型怎么优化部署？"
- "图像模型推理怎么加速？"
**回答框架**：
1. 先说结论：CV部署需要模型压缩、推理引擎优化和部署架构三管齐下
2. 展开原理：模型优化（ONNX转换+TensorRT加速、INT8量化、结构化剪枝）、推理加速（动态批处理、异步推理、模型预热、多实例负载均衡）、边缘部署（TFLite/Core ML、模型裁剪适配设备算力）、视频流处理（GPU解码、帧采样、管线并行）
3. 对比相关概念：CV部署偏计算优化，NLP/LLM部署偏内存和KV Cache优化
4. 举例应用场景：YOLOv8转TensorRT INT8，在Triton上服务，P99延迟<10ms，QPS提升5倍
**延伸考点**：模型压缩 → 模型服务 → 边缘部署

### LLM 推理服务
**核心要点**：LLM推理服务的核心是显存管理（KV Cache）、吞吐优化和弹性扩缩容
**常见面试问法**：
- "LLM推理服务怎么优化？"
- "vLLM的原理是什么？"
**回答框架**：
1. 先说结论：LLM推理的瓶颈是显存（KV Cache）和自回归生成的串行性
2. 展开原理：vLLM（PagedAttention管理KV Cache、虚拟内存思想消除碎片、连续批处理）、模型路由（大模型处理复杂请求、小模型处理简单请求、分类器路由）、弹性扩缩容（基于队列长度/GPU利用率自动扩缩、冷启动优化、模型预热）、推理优化（Flash Attention、Speculative Decoding、前缀缓存）、量化部署（GPTQ/AWQ/GGUF 4bit量化）
3. 对比相关概念：vLLM偏吞吐优化，TensorRT-LLM偏延迟优化，TGI偏易用性
4. 举例应用场景：用vLLM+PagedAttention部署LLaMA-7B，A100单卡吞吐从50 tok/s提升到2000 tok/s
**延伸考点**：模型服务 → 推理延迟优化 → GPU调度

## 十、项目高频追问

### 线上模型退化 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：线上模型退化是ML系统最常见的问题，需建立从发现到恢复的完整机制
**常见面试问法**：
- "线上模型效果下降了怎么办？"
- "你怎么发现模型退化的？怎么处理的？"
**回答框架**：
1. 先说结论：模型退化需从监控发现→根因分析→应急处理→长期预防形成闭环
2. 展开原理：发现（监控指标下降/数据漂移告警/用户反馈）、根因分析（数据问题/模型问题/工程问题/业务变化）、应急处理（回滚模型/降级策略/紧急重训）、长期预防（自动监控+重训管线、定期模型评估、数据质量门禁）
3. 对比相关概念：突发退化多为工程bug或数据异常，渐进退化多为概念漂移
4. 举例应用场景：推荐CTR突然下降10%，排查发现特征管线故障导致用户画像特征全为0，修复管线后回滚模型恢复
**延伸考点**：模型监控 → 模型重训练 → 部署策略

### 训练成本优化 [频率: ⭐⭐⭐⭐]
**核心要点**：训练成本优化从资源利用率、训练效率和架构选择三个维度降低成本
**常见面试问法**：
- "训练成本太高怎么优化？"
- "你们团队怎么控制GPU成本的？"
**回答框架**：
1. 先说结论：训练成本优化是资源效率、算法效率和工程效率的综合优化
2. 展开原理：资源利用率（GPU共享/弹性调度/Spot实例/自动缩容）、训练效率（混合精度/梯度累积/数据预取/编译优化）、架构选择（小模型蒸馏/高效架构/早停策略）、实验效率（超参搜索早停/小数据验证/低成本预实验）、存储优化（检查点压缩/数据格式选择/生命周期管理）
3. 对比相关概念：降成本不是降质量，而是消除浪费、提高效率
4. 举例应用场景：用Spot实例+自动检查点将训练成本降低60%，混合精度+编译优化将训练时间缩短40%
**延伸考点**：GPU调度 → 训练加速 → 分布式训练

### 推理延迟优化 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：推理延迟优化从模型、引擎、架构三个层面系统性降低P99延迟
**常见面试问法**：
- "模型推理太慢怎么优化？"
- "P99延迟怎么降？"
**回答框架**：
1. 先说结论：推理延迟优化需先定位瓶颈（计算/内存/IO），再针对性优化
2. 展开原理：模型层（量化INT8/INT4、剪枝、蒸馏小模型、知识蒸馏）、引擎层（TensorRT算子融合、ONNX Runtime图优化、Flash Attention、KV Cache优化）、架构层（动态批处理、异步推理、模型预热、多实例负载均衡、CDN边缘推理）、系统层（GPU直连、NUMA优化、内存池化）
3. 对比相关概念：模型层优化效果大但需重新训练，引擎层优化无需改模型，架构层优化提升并发能力
4. 举例应用场景：BERT推理P99 200ms→量化INT8→TensorRT优化→动态批处理→最终P99 15ms
**延伸考点**：模型压缩 → 模型服务 → LLM推理服务

### 数据安全与合规 [频率: ⭐⭐⭐⭐]
**核心要点**：ML系统的数据安全需覆盖数据采集、存储、训练、推理全链路
**常见面试问法**：
- "训练数据涉及用户隐私怎么处理？"
- "ML系统怎么做数据合规？"
**回答框架**：
1. 先说结论：ML数据安全需在数据全生命周期实施保护，满足隐私法规和业务合规要求
2. 展开原理：数据脱敏（匿名化/假名化/差分隐私）、访问控制（数据分级、RBAC、审计日志）、训练安全（联邦学习/安全多方计算/同态加密）、模型隐私（成员推理攻击防御、模型逆向防护）、合规框架（GDPR/个人信息保护法、数据出境、算法备案）
3. 对比相关概念：差分隐私是数学保证，联邦学习是架构保证，数据脱敏是流程保证
4. 举例应用场景：用户画像特征差分隐私加噪后训练，模型服务不返回原始特征只返回预测结果
**延伸考点**：模型治理 → 数据质量

### 多团队协作 [频率: ⭐⭐⭐]
**核心要点**：ML项目涉及数据、算法、工程、业务多团队，需明确接口和流程
**常见面试问法**：
- "ML项目多团队怎么协作？"
- "数据团队和算法团队怎么配合？"
**回答框架**：
1. 先说结论：多团队协作的核心是接口标准化、流程规范化和平台统一化
2. 展开原理：角色分工（数据团队负责数据质量和特征、算法团队负责模型研发、工程团队负责平台和部署、业务团队负责需求和指标）、接口标准化（特征接口、模型接口、数据接口的契约化）、流程规范化（需求评审→数据准备→模型开发→评估验收→上线发布）、平台统一化（统一特征平台、实验平台、部署平台减少协作摩擦）
3. 对比相关概念：小团队一人多角色效率高但难扩展，大团队分工明确但需强协作机制
4. 举例应用场景：特征平台统一数据团队和算法团队的接口，数据团队负责特征计算和SLA，算法团队负责特征使用和模型效果
**延伸考点**：特征存储 → 模型治理 → MLOps平台

# 测试开发面试八股文库

## 一、测试基础

### 测试分类 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：按阶段分为单元→集成→系统→验收，按方法分为黑盒/白盒/灰盒，不同分类维度对应不同的测试策略和工具选型。
**常见面试问法**：
- "请说说单元测试和集成测试的区别？"
- "黑盒测试和白盒测试分别适用于什么场景？"
**回答框架**：
1. 先按阶段分类说明：单元测试验证最小代码单元，集成测试验证模块交互，系统测试验证整体功能，验收测试验证业务需求
2. 再按方法分类：黑盒关注输入输出，白盒关注内部逻辑，灰盒介于两者之间
3. 举例说明：单元测试用pytest/JUnit，集成测试用Postman，系统测试用Selenium
4. 强调实际项目中如何组合使用，形成完整测试策略
**延伸考点**：测试金字塔 → 自动化测试策略

### 测试用例设计方法 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：等价类划分和边界值分析是最基础最常用的方法，正交法和场景法用于复杂业务逻辑覆盖。
**常见面试问法**：
- "给你一个登录页面，你怎么设计测试用例？"
- "等价类划分和边界值分析有什么区别？"
**回答框架**：
1. 先列出核心方法：等价类划分→边界值分析→正交法→场景法→错误推测法
2. 等价类划分：将输入域划分为有效等价类和无效等价类，减少用例数量
3. 边界值分析：重点测试边界及其附近值，因为边界处最易出错
4. 以登录功能为例：等价类划分用户名长度、边界值测试6位和18位密码、场景法覆盖正常登录和异常流程
**延伸考点**：正交实验法 → 测试用例优先级排序

### 缺陷生命周期 [频率: ⭐⭐⭐⭐]
**核心要点**：Bug从发现到关闭经历新建→确认→修复→验证→关闭的完整流程，优先级和严重程度是两个独立维度。
**常见面试问法**：
- "Bug的优先级和严重程度有什么区别？"
- "开发不认可你提的Bug怎么办？"
**回答框架**：
1. 定义严重程度：功能影响范围（致命/严重/一般/轻微）
2. 定义优先级：修复紧急程度（立即/高/中/低）
3. 说明两者关系：严重程度高不一定优先级高（如偶现问题），优先级受发布计划和影响面综合决定
4. 处理争议：提供复现步骤、日志截图，拉产品经理评估影响
**延伸考点**：Bug收敛分析 → 测试退出标准

### 回归测试与冒烟测试 [频率: ⭐⭐⭐⭐]
**核心要点**：冒烟测试验证核心功能是否可用，回归测试验证修改未引入新缺陷，两者目的和范围不同。
**常见面试问法**：
- "回归测试的策略是什么？怎么选择回归用例？"
- "冒烟测试和回归测试有什么区别？"
**回答框架**：
1. 冒烟测试：提测前/后快速验证核心功能，用例少、执行快，不通过则打回
2. 回归测试：代码变更后验证原有功能不受影响，范围根据变更影响面确定
3. 回归用例选择策略：全量回归（安全但耗时）、选择性回归（基于风险和影响分析）、自动化回归（持续集成中执行）
4. 实际项目中的实践：核心用例自动化做冒烟，全量用例自动化做回归
**延伸考点**：自动化测试金字塔 → CI/CD中的测试门禁

### 探索性测试 [频率: ⭐⭐⭐]
**核心要点**：探索性测试将测试设计和执行融合，依赖测试人员的经验和直觉，是对脚本化测试的重要补充。
**常见面试问法**：
- "什么是探索性测试？你平时怎么做？"
- "探索性测试和自由测试有什么区别？"
**回答框架**：
1. 定义：同时进行测试学习和测试设计的测试方法，不是随意测试
2. 与自由测试的区别：有目标、有思路、有记录，是结构化的探索
3. 常用方法：基于风险的探索、基于场景的探索、基于反馈的探索
4. 适用场景：需求不明确时、时间紧迫时、补充自动化无法覆盖的场景
**延伸考点**：测试策略制定 → 测试覆盖率评估

### 自动化测试金字塔 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：金字塔模型从底层到顶层依次是单元测试→接口测试→UI测试，越底层投入产出比越高，越顶层越脆弱但越贴近用户。
**常见面试问法**：
- "请说说自动化测试金字塔模型的理解？"
- "你们项目中自动化测试怎么分层的？"
**回答框架**：
1. 画出金字塔：底层单元测试（70%）→中层接口测试（20%）→顶层UI测试（10%）
2. 单元测试：速度快、稳定性高、维护成本低，但离用户远
3. 接口测试：兼顾效率和覆盖面，是性价比最高的层级
4. UI测试：最贴近用户但最脆弱、最慢、维护成本最高
5. 实际项目中根据团队情况调整比例，避免"冰淇淋蛋卷"反模式
**延伸考点**：接口自动化 → PO模式

## 二、自动化测试

### Selenium原理 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：Selenium通过WebDriver协议与浏览器驱动通信，浏览器驱动再操控浏览器执行操作，实现了跨浏览器的自动化控制。
**常见面试问法**：
- "请说说Selenium WebDriver的工作原理？"
- "Selenium 3和Selenium 4有什么区别？"
**回答框架**：
1. 架构组成：测试脚本→WebDriver API→浏览器驱动（ChromeDriver等）→浏览器
2. 通信机制：通过HTTP协议发送WebDriver标准命令，浏览器驱动将其转化为浏览器原生操作
3. Selenium 4升级：W3C标准协议替代JSON Wire Protocol、相对定位器、CDP支持
4. 优势：多语言支持、多浏览器支持、生态成熟；劣势：速度较慢、依赖浏览器驱动版本匹配
**延伸考点**：Playwright vs Selenium → 显式等待与隐式等待

### Playwright vs Cypress vs Selenium [频率: ⭐⭐⭐⭐]
**核心要点**：Playwright基于CDP多浏览器支持强，Cypress架构简单但仅支持Chromium，Selenium生态最成熟但速度最慢。
**常见面试问法**：
- "Playwright和Selenium有什么区别？你会选哪个？"
- "Cypress为什么只能测Chrome系浏览器？"
**回答框架**：
1. Selenium：WebDriver协议、多浏览器、多语言、生态成熟、速度慢
2. Cypress：直接运行在浏览器内、自动等待、调试方便、仅Chromium、不支持多标签页
3. Playwright：CDP+浏览器内注入、多浏览器、自动等待、并行执行、较新但增长快
4. 选型建议：新项目推荐Playwright，已有Selenium项目可继续，纯前端团队可考虑Cypress
**延伸考点**：Selenium原理 → 自动化框架选型

### PO模式（Page Object） [频率: ⭐⭐⭐⭐⭐]
**核心要点**：PO模式将页面元素定位和操作封装为页面对象类，实现测试逻辑与页面结构的解耦，提升用例可维护性。
**常见面试问法**：
- "什么是PO模式？为什么要用PO模式？"
- "PO模式中怎么处理页面间跳转？"
**回答框架**：
1. 定义：每个页面对应一个类，类中封装元素定位和操作方法
2. 核心原则：页面元素与测试逻辑分离、公共方法暴露操作接口、不暴露内部实现
3. 页面跳转：方法返回目标页面对象（return new HomePage()），实现链式调用
4. 进阶：BasePage封装公共操作（点击、输入、等待），子页面继承BasePage
**延伸考点**：关键字驱动 → 数据驱动测试

### 显式等待 vs 隐式等待 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：隐式等待全局生效设置一次即可，显式等待针对特定条件精确等待，两者不要混用。
**常见面试问法**：
- "显式等待和隐式等待有什么区别？"
- "为什么说两者不要混用？"
**回答框架**：
1. 隐式等待：driver.implicitly_wait()，全局生效，查找元素时轮询DOM，超时抛异常
2. 显式等待：WebDriverWait+expected_conditions，针对特定条件（可见、可点击等）精确等待
3. 混用问题：隐式等待和显式等待叠加会导致等待时间不可预测
4. 最佳实践：只用显式等待，或只用隐式等待+Thread.sleep处理特殊场景
**延伸考点**：Selenium原理 → 自动化用例稳定性

### 关键字驱动与数据驱动 [频率: ⭐⭐⭐⭐]
**核心要点**：数据驱动将测试数据与逻辑分离，关键字驱动将操作封装为关键字，两者结合可实现非编程人员编写用例。
**常见面试问法**：
- "数据驱动和关键字驱动有什么区别？"
- "你们项目中怎么实现数据驱动的？"
**回答框架**：
1. 数据驱动：同一测试逻辑，不同测试数据（CSV/Excel/YAML/数据库），用@pytest.mark.parametrize实现
2. 关键字驱动：将操作封装为"打开页面""输入文本""点击按钮"等关键字，通过表格组合用例
3. 数据驱动关注"测什么"，关键字驱动关注"怎么测"
4. 实际项目：pytest参数化做数据驱动，PO模式+关键字封装做关键字驱动
**延伸考点**：PO模式 → 自动化测试框架设计

### HTTP协议与RESTful API [频率: ⭐⭐⭐⭐⭐]
**核心要点**：HTTP是无状态的应用层协议，RESTful API基于HTTP方法语义化设计接口（GET查/POST增/PUT改/DELETE删）。
**常见面试问法**：
- "HTTP和HTTPS有什么区别？"
- "GET和POST请求有什么区别？"
**回答框架**：
1. HTTP基础：请求方法（GET/POST/PUT/DELETE）、状态码（2xx成功/3xx重定向/4xx客户端错误/5xx服务端错误）
2. GET vs POST：GET参数在URL中、有长度限制、可缓存；POST参数在Body中、无限制、不可缓存
3. RESTful设计：资源用名词、操作用HTTP方法、无状态、统一接口
4. HTTPS = HTTP + TLS加密，端口443 vs 80
**延伸考点**：接口自动化 → Mock服务

### 接口自动化测试 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：接口自动化验证API的功能、性能和安全性，核心是参数化构造请求、断言响应、处理接口间关联。
**常见面试问法**：
- "接口自动化测试怎么做的？"
- "接口之间有依赖怎么处理？"
**回答框架**：
1. 工具选型：requests+pytest（灵活）、Postman（快速）、JMeter（压测+接口）
2. 核心流程：构造请求→发送请求→断言响应→记录结果
3. 关联处理：上一个接口的响应提取为变量，传入下一个接口（token传递、ID传递）
4. 进阶能力：数据驱动、签名鉴权、加密解密、文件上传下载、Mock服务
**延伸考点**：HTTP协议 → Mock服务

### Mock服务 [频率: ⭐⭐⭐⭐]
**核心要点**：Mock服务模拟未开发完成或不可用的依赖接口，使测试不依赖外部环境，提高用例稳定性和执行效率。
**常见面试问法**：
- "为什么要用Mock？怎么实现？"
- "Mock会不会导致真实问题被掩盖？"
**回答框架**：
1. 使用场景：依赖接口未开发、第三方接口不稳定、测试异常场景（超时、错误码）
2. 实现方式：代码级Mock（unittest.mock/Mockito）、工具级Mock（MockServer/Moco/Postman Mock）
3. 风险控制：Mock和真实接口需保持一致、上线前必须用真实接口验证、定期校验Mock数据
4. 最佳实践：开发阶段用Mock提效，回归阶段用真实接口保障
**延伸考点**：接口自动化 → 微服务测试策略

### Appium原理 [频率: ⭐⭐⭐⭐]
**核心要点**：Appium基于WebDriver协议，通过Bootstrap（Android）或XCUITest（iOS）驱动移动端UI操作，跨平台且不修改App源码。
**常见面试问法**：
- "请说说Appium的工作原理？"
- "Appium在Android和iOS上的驱动有什么区别？"
**回答框架**：
1. 架构：测试脚本→Appium Server→UIAutomator2（Android）/XCUITest（iOS）→设备
2. Android：Appium注入Bootstrap.jar到被测App，通过UIAutomator执行命令
3. iOS：Appium启动WebDriverAgent，通过XCUITest框架执行命令
4. 优势：跨平台、多语言、不修改源码；劣势：速度较慢、环境配置复杂
**延伸考点**：Selenium原理 → 混合App测试

### 混合App测试（WebView） [频率: ⭐⭐⭐]
**核心要点**：混合App包含原生和WebView两部分，测试时需切换上下文（Context）分别操控原生控件和H5页面。
**常见面试问法**：
- "混合App怎么自动化测试？"
- "WebView中的元素怎么定位？"
**回答框架**：
1. 识别混合App：原生框架+内嵌WebView加载H5页面
2. 上下文切换：driver.contexts获取所有上下文，切换到WEBVIEW_xxx操控H5元素
3. 调试前提：Android需开启WebView调试模式（setWebContentsDebuggingEnabled）
4. 常见问题：上下文切换失败、H5页面加载慢、跨域问题
**延伸考点**：Appium原理 → 移动端自动化框架选型

## 三、性能测试

### JMeter核心组件 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：JMeter通过线程组模拟并发用户，配合定时器控制节奏、取样器发送请求、断言验证结果、监听器收集数据。
**常见面试问法**：
- "JMeter的核心组件有哪些？各自的作用？"
- "JMeter中线程组怎么配置？"
**回答框架**：
1. 线程组：设置并发用户数、Ramp-Up时间、循环次数
2. 取样器：HTTP请求、JDBC请求等，模拟用户操作
3. 定时器：控制请求间隔（Constant Timer、Gaussian Random Timer）
4. 断言：验证响应内容（响应码、JSON Path、持续时间）
5. 监听器：查看结果树、聚合报告、图形结果
**延伸考点**：压测方案设计 → 性能瓶颈定位

### 并发数/QPS/TPS/响应时间 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：并发数是同时在线用户数，QPS是每秒查询数，TPS是每秒事务数，响应时间是从请求到响应的耗时，四者共同描述系统性能。
**常见面试问法**：
- "QPS和TPS有什么区别？"
- "并发数和QPS是什么关系？"
**回答框架**：
1. 并发数：系统同时处理的请求数（并发用户数≠在线用户数）
2. QPS：每秒查询数，衡量读接口吞吐量
3. TPS：每秒事务数，衡量包含读写的完整业务吞吐量
4. 响应时间：RT = 网络时间+服务处理时间+数据库时间，关注P50/P90/P99
5. 关系：QPS = 并发数 / 平均响应时间（Little's Law）
**延伸考点**：压测方案设计 → 性能调优

### 压测方案设计 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：压测方案需明确测试目标、场景设计、数据准备、监控指标和通过标准，核心是模拟真实用户行为。
**常见面试问法**：
- "你做过性能测试吗？方案怎么设计的？"
- "压测场景有哪些？"
**回答框架**：
1. 明确目标：确定要测的指标（QPS、RT、成功率）和通过标准
2. 场景设计：基准测试（单用户）→负载测试（逐步加压）→压力测试（超负荷）→稳定性测试（长时间）
3. 数据准备：铺底数据量与生产一致、参数化测试数据、预热缓存
4. 执行与监控：逐步加压观察拐点、监控服务端资源（CPU/内存/IO）、记录瓶颈点
**延伸考点**：性能瓶颈定位 → 性能调优

### 性能瓶颈定位 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：性能瓶颈定位遵循从外到内、从上到下的原则，先确定瓶颈层（网络→应用→数据库→系统），再深入分析根因。
**常见面试问法**：
- "性能测试发现接口慢，你怎么定位？"
- "CPU飙高怎么排查？"
**回答框架**：
1. 确定瓶颈层：网络（带宽/延迟）→应用（CPU/内存/线程）→数据库（慢SQL/锁）→系统（IO/磁盘）
2. CPU飙高：top定位进程→top -Hp定位线程→jstack分析线程栈→定位热点代码
3. 内存问题：jmap导出堆dump→MAT分析大对象/泄漏→定位代码
4. 慢SQL：慢查询日志→EXPLAIN分析执行计划→优化索引/SQL
5. IO瓶颈：iostat查看磁盘利用率→优化写入策略/增加缓存
**延伸考点**：压测方案设计 → 性能调优思路

### 性能调优思路 [频率: ⭐⭐⭐⭐]
**核心要点**：性能调优遵循"先定位后优化"原则，从最易见效的点入手，每次只调一个变量，验证效果后再调下一个。
**常见面试问法**：
- "你做过哪些性能调优？效果如何？"
- "性能调优的优先级怎么排？"
**回答框架**：
1. 调优原则：先定位瓶颈→先优化效果最大的→每次只改一个变量→用数据说话
2. 常见优化手段：加缓存（Redis）→优化SQL（索引/执行计划）→异步化（MQ）→连接池调优→代码优化
3. 数据库优化：索引优化→SQL改写→分库分表→读写分离
4. 应用优化：线程池调优→JVM参数调优→减少锁竞争→异步非阻塞
**延伸考点**：性能瓶颈定位 → 缓存策略

3. 核心语法：pipeline{agent/stages/steps/post}，stage划分阶段
4. 实践：多分支Pipeline、共享库、参数化构建、条件执行

### Git工作流 [频率: ⭐⭐⭐⭐]
**核心要点**：Git Flow适合版本发布节奏慢的项目，Trunk Based适合持续部署的团队，选择取决于团队发布模式。

3. GitHub Flow：main+feature分支，PR合并，适合持续部署
4. 选型：发布周期长选Git Flow，持续部署选Trunk Based/GitHub Flow
**延伸考点**：Jenkins Pipeline → 代码评审流程

### Docker基础 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：Docker通过容器技术实现应用与运行环境的打包，镜像只读分层存储，容器是镜像的运行实例。
**常见面试问法**：
- "Docker镜像和容器有什么区别？"

2. Dockerfile指令：FROM基础镜像→RUN执行命令→COPY/ADD复制文件→WORKDIR工作目录→EXPOSE暴露端口→ENTRYPOINT/CMD启动命令
3. 镜像分层：每条指令生成一层，相同层可复用，减小镜像体积
4. 多阶段构建：编译阶段+运行阶段分离，最终镜像只包含运行时依赖
**延伸考点**：Docker网络模式 → K8s Pod

### 持续集成/持续交付流程 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：CI是频繁合并代码并自动构建测试，CD是自动将通过测试的代码部署到生产，核心是自动化和快速反馈。
**常见面试问法**：

3. CD（持续部署）：CI通过→自动部署到生产，无需人工干预
4. 关键实践：自动化测试门禁、代码评审、特性开关、蓝绿/金丝雀发布
**延伸考点**：Jenkins Pipeline → 灰度发布策略

### 代码质量：SonarQube [频率: ⭐⭐⭐]
**核心要点**：SonarQube是静态代码分析平台，集成到CI流水线中自动检测代码异味、安全漏洞和重复代码。
**常见面试问法**：

**延伸考点**：CI/CD流程 → 代码评审

## 五、编程基础

### Python装饰器 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：装饰器是接受函数并返回新函数的高阶函数，用于在不修改原函数代码的情况下增强功能，本质是语法糖。
**常见面试问法**：
- "请说说Python装饰器的原理？"
- "手写一个带参数的装饰器"
**回答框架**：
1. 本质：@decorator等价于func = decorator(func)，是语法糖
2. 无参装饰器：外层函数接收被装饰函数，内层函数添加增强逻辑，返回内层函数
3. 带参装饰器：最外层接收装饰器参数，中间层接收被装饰函数，最内层执行增强逻辑
4. functools.wraps：保留原函数元信息（__name__、__doc__）
5. 应用场景：日志记录、权限校验、性能计时、重试机制
**延伸考点**：Python生成器 → pytest fixture

### Python生成器 [频率: ⭐⭐⭐⭐]
**核心要点**：生成器通过yield关键字实现惰性求值，只在需要时产生值，节省内存，适合处理大数据集。
**常见面试问法**：
- "yield和return有什么区别？"
- "生成器有什么应用场景？"
**回答框架**：
1. yield vs return：return返回结果并终止函数，yield暂停函数并保留状态，下次调用从暂停处继续
2. 惰性求值：不一次性生成所有数据，按需生成，内存占用恒定
3. 创建方式：生成器函数（yield）、生成器表达式（(x for x in range(10))）
4. 应用场景：读取大文件逐行处理、无限序列、协程基础
**延伸考点**：装饰器 → 迭代器协议

### pytest框架 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：pytest是Python最流行的测试框架，支持fixture灵活管理测试前置条件，参数化实现数据驱动，插件生态丰富。
**常见面试问法**：
- "pytest和unittest有什么区别？"
- "pytest fixture怎么用？"
**回答框架**：
1. pytest vs unittest：pytest无需类继承、断言更简洁（assert）、fixture比setUp/tearDown更灵活
2. fixture：@pytest.fixture定义，scope控制作用域（function/class/module/session），yield实现teardown
3. 参数化：@pytest.mark.parametrize实现数据驱动，多组数据自动生成多条用例
4. 插件：pytest-html报告、allure美化报告、pytest-xdist并行执行、pytest-rerunfailures失败重跑
**延伸考点**：数据驱动 → 自动化测试框架设计

### Linux常用命令 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：测试开发需掌握文件操作（grep/awk/sed/find）、进程管理（top/ps/kill）、网络排查（netstat/curl/telnet）三大类命令。
**常见面试问法**：
- "怎么查看某个进程的CPU和内存占用？"
- "怎么实时查看日志文件？"
**回答框架**：
1. 文件操作：grep过滤、awk列处理、sed行编辑、find查找文件
2. 进程管理：ps aux查看进程、top实时监控、kill发送信号
3. 网络排查：netstat查看端口、curl测试接口、telnet测试连通性
4. 日志分析：tail -f实时查看、grep过滤错误、awk统计频次
**延伸考点**：Shell脚本 → 性能瓶颈定位

### SQL查询 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：测试开发需掌握JOIN连接查询、子查询、聚合分组，以及索引原理用于验证数据正确性和辅助性能测试。
**常见面试问法**：
- "LEFT JOIN和INNER JOIN有什么区别？"
- "索引有哪些类型？什么情况下索引会失效？"
**回答框架**：
1. JOIN：INNER JOIN取交集、LEFT JOIN保留左表全部、RIGHT JOIN保留右表全部
2. 子查询：WHERE中嵌套SELECT，IN/EXISTS/ANY/ALL
3. 聚合分组：GROUP BY分组+HAVING过滤，配合COUNT/SUM/AVG/MAX/MIN
4. 索引：B+树索引、唯一索引、联合索引（最左前缀原则）；失效场景：函数操作、隐式转换、LIKE左模糊、OR条件
**延伸考点**：数据库性能调优 → 数据驱动测试

### 网络基础（TCP/HTTP/DNS） [频率: ⭐⭐⭐⭐⭐]
**核心要点**：TCP提供可靠传输（三次握手/四次挥手），HTTP是应用层协议，DNS负责域名解析，三者是网络通信的基础。
**常见面试问法**：
- "TCP三次握手和四次挥手的过程？"
- "HTTP状态码有哪些？301和302的区别？"
**回答框架**：
1. TCP三次握手：SYN→SYN+ACK→ACK，建立连接
2. TCP四次挥手：FIN→ACK→FIN→ACK，断开连接（TIME_WAIT等待2MSL）
3. HTTP：无状态协议，1.0短连接→1.1长连接→2.0多路复用→3.0基于QUIC
4. DNS：域名→IP的递归/迭代查询过程，浏览器缓存→系统缓存→本地DNS→根DNS
**延伸考点**：HTTP协议 → 接口自动化

### 不稳定用例（Flaky Test）处理 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：Flaky Test是自动化测试最大痛点，需从根因分析（时序问题/环境依赖/数据竞争）入手，而非简单重试。
**常见面试问法**：
- "怎么处理不稳定用例？"
- "Flaky Test比例多少可以接受？"
**回答框架**：
1. 根因分类：时序问题（等待不够）、环境依赖（网络/服务不稳定）、数据竞争（用例间耦合）
2. 解决策略：显式等待替代隐式等待、数据隔离（每条用例独立数据）、幂等设计
3. 短期方案：失败重跑机制（pytest-rerunfailures），标记Flaky用例单独分析
4. 长期方案：建立Flaky Test监控看板，持续治理，目标Flaky率<5%
**延伸考点**：显式等待 → 自动化测试框架

### 重大Bug定位 [频率: ⭐⭐⭐⭐⭐]
**核心要点**：考察Bug定位的系统性思维，从现象→日志→代码→根因的完整链路，体现技术深度和业务理解。

3. 修复验证：开发修复→回归验证→补充用例防止复发

**延伸考点**：缺陷生命周期 → 测试策略
